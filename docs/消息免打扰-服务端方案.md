# 消息免打扰 — 服务端方案（详细）

本文从**业务含义、数据模型、接口设计、推送/通知逻辑、多端同步与扩展**几方面，说明「消息免打扰」在服务端如何设计与实现。

---

## 一、业务含义与影响范围

### 1.1 免打扰在服务端的含义

用户对**某个会话**（群或单聊）开启「消息免打扰」后，服务端需要让该会话的**新消息**对用户「少打扰」或「不打扰」，具体可体现在：

| 维度 | 开启免打扰后 | 说明 |
|------|----------------|------|
| **推送（Push）** | 不发推送，或只发静音推送 | 不弹窗、不亮屏、不响铃；或仅更新角标/静音通道 |
| **声音 / 振动** | 不触发提示音、不振动 | 由客户端根据服务端下发的「是否免打扰」决定 |
| **角标 / 红点** | 可选：仍计未读数但弱化，或不计入总未读 | 策略可配置：免打扰会话未读是否参与 App 角标、是否在列表显示红点 |
| **列表排序 / 提醒** | 可选：免打扰会话不置顶、不单独提醒 | 部分产品免打扰群仍显示未读数，但不参与「有人@我」等强提醒 |

**核心**：服务端要**持久化**「谁对哪个会话开启了免打扰」，并在**下发消息、推送、未读统计**时使用这份配置。

### 1.2 与前端的关系

- **前端**：提供开关 UI，调用服务端「设置免打扰」接口，并拉取「当前用户免打扰列表」用于列表/角标展示。
- **服务端**：存配置、在推送与未读逻辑里读配置，并可选在**消息体或会话元数据**里带「该会话对当前用户是否免打扰」，供多端一致展示。

---

## 二、数据模型

### 2.1 核心表：用户会话设置（按会话维度）

以「用户 + 会话」为维度存免打扰状态，便于按会话查询、按用户批量拉取。

**表名**：`user_conversation_settings` 或 `user_chat_mute`（按项目命名习惯即可）

| 字段 | 类型 | 说明 |
|------|------|------|
| `user_id` | string / bigint | 用户 ID |
| `conversation_type` | tinyint / enum | 会话类型：1 单聊 2 群聊 |
| `conversation_id` | string / bigint | 会话 ID（单聊可为对方 userId，群聊为 groupId） |
| `muted` | boolean / tinyint | 是否免打扰：0 否 1 是 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

**唯一约束**：`(user_id, conversation_type, conversation_id)`，保证每个用户每个会话只有一条配置。

**索引建议**：

- 主键或唯一：`(user_id, conversation_type, conversation_id)`
- 若按用户拉取全量免打扰列表：`user_id` 单列索引即可

**示例**：

```text
user_id   conversation_type   conversation_id   muted   updated_at
user_1    2                    group_101         1       2025-02-20 10:00:00
user_1    1                    user_2            0       2025-02-19 08:00:00
```

表示：用户 user_1 对群 group_101 开启免打扰，对单聊 user_2 未开启。

### 2.2 可选：全局免打扰

若产品有「全局免打扰时段」或「总开关」，可单独表或在本表加会话维度的特殊值，例如：

- 方案 A：单独表 `user_global_settings`，字段如 `mute_all_until`（时间戳）、`mute_all`（bool）
- 方案 B：在 `user_conversation_settings` 里用特殊 `conversation_id`（如 `"__all__"`）表示全局设置

下文仅讨论**按会话免打扰**，全局逻辑可在此基础上叠加。

## 三、接口设计

### 3.1 设置免打扰（写）

**接口**：`POST /api/chat/mute` 或 `PUT /api/user/conversation/settings`

**鉴权**：需要登录，识别当前 `user_id`。

**请求体**：

```json
{
  "conversation_type": "group",
  "conversation_id": "group_101",
  "muted": true
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| conversation_type | string | 是 | `c2c` / `group` |
| conversation_id | string | 是 | 单聊为对方 userId，群聊为 groupId |
| muted | boolean | 是 | true 开启免打扰，false 关闭 |

**响应**：

```json
{
  "ok": true,
  "data": {
    "conversation_type": "group",
    "conversation_id": "group_101",
    "muted": true
  }
}
```

**服务端逻辑**：

1. 校验用户是否在该会话中（单聊需存在双方关系，群聊需在群成员表）。
2. `upsert`：存在则更新 `muted`、`updated_at`，不存在则 INSERT。
3. 可选：发一条配置同步消息（见下「多端同步」）或仅依赖下次拉取。

### 3.2 查询免打扰列表（读）

**接口**：`GET /api/chat/mute/list` 或 `GET /api/user/conversation/settings?keys=muted`

**鉴权**：需要登录。

**请求**：可无 body；或 `?conversation_type=group` 只查群聊。

**响应**：

```json
{
  "ok": true,
  "data": {
    "muted_conversations": [
      { "conversation_type": "group", "conversation_id": "group_101" },
      { "conversation_type": "c2c", "conversation_id": "user_2" }
    ]
  }
}
```

或直接返回 ID 列表，便于前端做 `Set` 判断：

```json
{
  "ok": true,
  "data": {
    "muted_group_ids": ["group_101"],
    "muted_c2c_user_ids": ["user_2"]
  }
}
```

**服务端逻辑**：按 `user_id` 查 `user_conversation_settings` 且 `muted = 1`，按类型分组或原样返回。

### 3.3 单会话是否免打扰（按需）

若单条消息或会话详情接口需要带「当前用户对该会话是否免打扰」，可在：

- **会话列表接口**：每个会话多返回 `muted: boolean`
- **单会话详情 / 拉取会话设置接口**：`GET /api/chat/conversation/:type/:id/settings` 返回 `{ muted: true }`

实现均为：用当前 `user_id` + `conversation_type` + `conversation_id` 查 `user_conversation_settings`，取 `muted`。

## 四、推送与通知逻辑（核心）

### 4.1 流程概览

1. 新消息落库并投递到接收方在线连接（WebSocket / 长连接）。
2. 对**每个接收方**判断：该会话对该用户是否免打扰。
3. 根据结果决定：**是否发推送、是否静音、角标如何计算**。

### 4.2 推送是否下发

**伪代码**：

```text
on_new_message(message, recipient_user_id):
  conv_type = message.conversation_type
  conv_id   = message.conversation_id

  muted = get_user_conversation_muted(recipient_user_id, conv_type, conv_id)
  if muted:
    # 策略 A：完全不发推送
    return  # 不调用 Push 服务
  # 策略 B：发静音推送（仅更新角标、不响铃）
  # send_push(recipient_user_id, message, silent=true)
  else:
    send_push(recipient_user_id, message, silent=false)
```

- **get_user_conversation_muted**：查 `user_conversation_settings` 表，`(user_id, conversation_type, conversation_id)` 且 `muted = 1`。
- 若希望「免打扰仍收静音推送」，则 `muted` 时也调推送网关，但带 `silent=true` 或等价参数，由推送通道/客户端不响铃不振动。

### 4.3 推送内容与静音参数

- **静音推送**：FCM / APNs 等均有「静音」或「content-available」类参数，服务端在组推送 payload 时根据 `muted` 设置。
- **角标**：若免打扰会话**不计入** App 角标，则服务端在「未读数/角标」接口中排除免打扰会话的未读；若**计入**，则推送里角标数可包含该会话，仅不响铃。

### 4.4 未读数与角标

- **未读表**：每条消息的已读状态、未读数汇总可按现有设计，与是否免打扰解耦。
- **角标数**：若产品定义为「免打扰会话不参与角标」，则：
  - 角标接口：`未读数 = SUM(未读) WHERE 会话 NOT IN (当前用户免打扰列表)`；
  - 或：未读照常算，角标接口单独返回 `badge_count` 和 `muted_unread_count`，由客户端决定角标用哪个。

## 五、多端同步

### 5.1 问题

用户在 A 端把某群设为免打扰，B 端（其他设备/Web）应尽快看到「免打扰已开启」，列表与角标一致。

### 5.2 方案一：拉取时带配置（简单）

- 各端在**拉取会话列表**或**拉取用户设置**时，服务端返回每个会话的 `muted`，或返回 `muted_conversation_ids`。
- 前端本地更新 store，无需额外推送。缺点：有延迟，依赖下次拉取。

### 5.3 方案二：配置变更推送（实时）

- 用户调用「设置免打扰」接口成功后，服务端向该用户**所有在线端**发一条**配置同步**消息（或走现有信令通道）。
- 消息体示例：`{ type: "conversation_settings", conversation_type, conversation_id, muted }`。
- 各端收到后更新本地缓存或再调一次「免打扰列表」接口，保证多端一致。

### 5.4 方案三：多端存储

- 配置只存服务端，各端不持久化；每次启动/拉会话列表时从服务端拿最新。
- 或：端上可缓存一份，但以服务端为准，拉取时覆盖。

## 六、扩展与边界

### 6.1 单聊免打扰

- 数据模型同上，`conversation_type = c2c`，`conversation_id` 为对方 `user_id`。
- 推送与未读逻辑同样：先查该用户对该单聊是否 `muted`，再决定是否推送、是否静音。

### 6.2 @ 提醒与免打扰

- 部分产品：免打扰群仍接收「有人@我」的强提醒。实现方式：
  - 推送前多一步判断：若消息内容或扩展字段里包含 @ 当前用户，则即使用户对该群免打扰，也发**非静音**推送（或单独「@ 提醒」通道）。
  - 需在消息体或扩展里标出 at 了谁，或单独 at 列表字段。

### 6.3 群主/管理员强制提醒

- 若存在「群主发全员通知不受免打扰限制」，可在推送逻辑里对「发送者身份 + 消息类型」做分支，再决定是否忽略 `muted`。

### 6.4 性能与缓存

- **读多写少**：可对 `user_conversation_settings` 做缓存（按 user_id 缓存其全部 muted 列表），设置时写 DB 并失效该用户缓存。
- **推送路径**：推送前查 muted 的 QPS 可能较高，缓存 key 如 `mute:${userId}`，value 为 `Set<conversation_type:conversation_id>` 或序列化列表，设置时更新缓存。

---

## 七、小结表

| 项目 | 建议 |
|------|------|
| **存储** | 表 `user_conversation_settings`，维度 (user_id, conversation_type, conversation_id)，字段 muted + 时间戳 |
| **写接口** | POST/PUT 设置单会话 muted，鉴权后 upsert |
| **读接口** | GET 拉取当前用户免打扰列表（或会话列表带 muted） |
| **推送** | 新消息下发前查 muted，为 true 则不发推送或发静音推送 |
| **角标** | 按产品决定免打扰会话是否计入角标，在未读/角标接口中过滤或单独返回 |
| **多端** | 拉取时带 muted，或配置变更时推配置同步消息 |
| **扩展** | @ 提醒、管理员强提醒可在推送分支里特殊处理；muted 查询可缓存 |

按上述模型与流程实现后，服务端即可完整支持「消息免打扰」的持久化与推送/角标策略，并与前端现有免打扰开关与展示逻辑对接。
