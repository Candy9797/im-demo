# 未读消息计数维护

本文档描述 IM 系统中未读消息计数的维护方式与数据流。

---

## 一、场景与定义

### 1.1 两类未读

| 场景 | 数据来源 | 定义 |
|------|----------|------|
| **客服 IM（Help & Support）** | chatStore | 单会话：Bot/Agent 发来且 `status !== read` 的消息数 |
| **会话页（/chat 好友/群组）** | chatSessionStore | 多会话：每个好友/群组的 `unreadCount` 字段 |

### 1.2 未读判定规则

- **客服 IM**：`senderType !== USER && senderType !== SYSTEM && status !== "read"`
  - 即：来自 Bot 或 Agent 且未被标记已读的消息
- **好友/群组**：按会话维度存储在 `Friend.unreadCount` / `Group.unreadCount`
  - 当前为 Mock 数据，无实时更新逻辑

## 二、客服 IM 未读维护

### 2.1 计数计算

**位置**：`src/store/chatStore.ts` — `countUnread(messages, userId)`

```typescript
export function countUnread(messages: Message[], userId: string | undefined): number {
  if (!userId) return 0;
  return messages.filter(
    (m) =>
      m.senderType !== SenderType.USER &&
      m.senderType !== SenderType.SYSTEM &&
      m.status !== "read"
  ).length;
}
```

- 未读数 = 消息列表中「非用户、非系统」且 `status !== "read"` 的数量
- 不持久化，每次从当前 `messages` 实时计算

### 2.2 标记已读触发

**触发点**：消息进入视口

**流程**：
```
MessageList 渲染 MessageItem
    │
    ▼
MessageItem 接收 onVisible 回调（来自 markAsRead）
    │
    ▼
IntersectionObserver 监听，threshold: 0.5
    │
    ▼
消息 50% 进入视口 → onVisible(messageId) 被调用
    │
    ▼
chatStore.markAsRead([messageId])
    │
    ├─ client.markAsRead(messageIds) → 发送 MARK_READ 帧
    │
    └─ （服务端返回 READ_RECEIPT 后，chatStore 更新消息 status）
```

**注意**：仅对「非用户消息」做已读上报（MessageItem 中 `isUser` 时不同步调用 `onVisible`）。

### 2.3 已读状态同步

**客户端 → 服务端**：`markAsRead(messageIds)` → `FrameType.MARK_READ`

**服务端**（`server/ws-handler.ts`）：
1. `db.updateMessageStatus(msgId, "read")`
2. 更新 `metadata.readBy`，加入当前 userId
3. 向该用户发送 `FrameType.READ_RECEIPT { messageIds, readBy }`

**客户端收到 READ_RECEIPT**：
```typescript
// chatStore 订阅 READ_RECEIPT
messages: s.messages.map((m) =>
  messageIds.includes(m.id)
    ? { ...m, status: "read", metadata: { ...meta, readBy: [...readByList, readBy] } }
    : m
)
```

- 更新对应消息的 `status` 和 `metadata.readBy`
- 下次 `countUnread` 时，这些消息不再计入未读

### 2.4 展示位置

- **ChatWindow**：头部、最小化栏的未读角标
- 调用 `countUnread(messages, auth?.userId)` 计算

## 三、会话页（好友/群组）未读维护

### 3.1 数据模型

**Friend / Group**（`src/lib/chatSessionMock.ts`）：
```typescript
interface Friend {
  unreadCount: number;  // 该好友会话的未读数
  // ...
}
interface Group {
  unreadCount: number;  // 该群组的未读数
  // ...
}
```

### 3.2 清零逻辑

**位置**：`src/store/chatSessionStore.ts` — `clearUnread`

- `selectFriend(friend)` 时：`clearUnread('c2c', friend.id)` → `friend.unreadCount = 0`
- `selectGroup(group)` 时：`clearUnread('group', group.id)` → `group.unreadCount = 0`

### 3.3 当前限制

- 未读数来自 Mock 初始值，无实时增长逻辑
- 进入会话即清零，不依赖消息 `status` 或服务端同步
- 若接入真实后端，需在收到新消息时按会话增加 `unreadCount`，进入会话或拉取已读状态时清零

## 四、数据流示意

### 4.1 客服 IM 未读全流程

```
Bot/Agent 发消息 → MESSAGE 帧 → messages 新增一条 (status: delivered)
    │
    ▼
countUnread 增加 1
    │
    ▼
用户滚动使消息进入视口 → IntersectionObserver 触发
    │
    ▼
markAsRead([msgId]) → MARK_READ 帧
    │
    ▼
服务端更新 DB status、readBy → 发送 READ_RECEIPT
    │
    ▼
chatStore 更新 messages 中对应项 status = "read"
    │
    ▼
countUnread 减少 1
```

### 4.2 好友/群组未读（Mock）

```
进入会话页 → 读取 friends[].unreadCount、groups[].unreadCount
    │
    ▼
点击好友/群组 → selectFriend / selectGroup → clearUnread
    │
    ▼
该会话 unreadCount = 0
```

## 五、扩展方向

| 场景 | 扩展点 |
|------|--------|
| 客服 IM | 服务端持久化「已读进度」，多端同步；离线期间未读在重连后恢复 |
| 好友/群组 | 新消息推送时按会话累加 unreadCount；进入会话时上报已读并清零 |
| 多会话 | 会话列表按会话维度维护未读，支持总未读、按会话未读 |
| 免打扰 | 支持会话级免打扰，免打扰会话不参与总未读角标 |
