# 客服 IM 系统 - 技术方案（详细版）

## 1. 项目概述与目标

### 1.1 项目定位

面向 Web3 交易所的**客服即时通讯（IM）Demo**，参考 Binance 客服体系，实现从智能客服（Bot）到人工客服（Agent）的完整对话链路，具备生产级架构与高 QPS、大消息等扩展能力。

### 1.2 核心目标

| 目标 | 说明 |
|------|------|
| **实时性** | WebSocket 长连接、乐观更新、心跳保活、断线重连 |
| **可扩展** | 协议支持 JSON/Protobuf 双格式、大消息分片、序列号与去重 |
| **高 QPS** | 消息批处理、限流、虚拟列表预留、序列号排序与去重 |
| **可维护** | SDK 与 UI 解耦、事件驱动、类型完备、文档与注释 |

### 1.3 技术栈

- **前端**：Next.js 16（App Router）、React 19、TypeScript、Zustand
- **后端**：Node.js、Express、ws（WebSocket）、better-sqlite3
- **SDK**：自研 IM SDK（无第三方 IM 依赖），支持 JSON/Protobuf、分片传输

## 3. 协议设计

### 3.1 WebSocket 帧结构（统一封装）

所有 C2S / S2C 消息均封装为**帧（Frame）**：

```typescript
interface Frame {
  type: FrameType;   // 帧类型，见下表
  seq: number;      // 单调递增序列号，用于排序、去重、断点续传
  timestamp: number; // 毫秒时间戳
  payload: unknown;  // 根据 type 解析的业务负载
}
```

**序列号作用**：

- **顺序**：保证展示顺序与发送顺序一致  
- **去重**：网络重试导致重复帧时可识别  
- **空洞检测**：发现丢包可请求 sync/history 补全  
- **幂等**：同一 seq 可安全重试  

### 3.2 帧类型一览（FrameType）

| 方向 | 类型 | 说明 | payload 典型结构 |
|------|------|------|------------------|
| C→S | `auth` | 认证（实际以 URL ?token= 为主） | - |
| C→S | `send_message` | 发送消息 | `Message` 或 `Message[]` |
| C→S | `typing` | 正在输入 | `{ senderType }` |
| C→S | `ack` | 消息确认 | - |
| C→S | `ping` | 心跳 Ping | `{ ts }` |
| C→S | `request_agent` | 请求转人工 | - |
| C→S | `sync` | 同步（拉取 afterSeqId 之后） | `{ afterSeqId }` |
| C→S | `load_history` | 拉取历史（beforeSeqId 之前） | `{ beforeSeqId }` |
| C→S | `mark_read` | 标记已读 | `{ messageIds }` |
| C→S | `add_reaction` / `remove_reaction` | 表情反应 | `{ messageId, emoji }` |
| C→S | `edit_message` / `recall_message` | 编辑/撤回 | `{ messageId, content? }` |
| S→C | `auth_ok` | 认证成功 | `{ conversationId, messages, hasMore, phase, ... }` |
| S→C | `message` | 收到消息（单条或批量） | `Message` 或 `Message[]` |
| S→C | `message_ack` | 消息送达确认 | `{ clientMsgId, serverMsgId, seqId }[]` |
| S→C | `typing_start` / `typing_stop` | 输入状态 | `{ senderType }` |
| S→C | `pong` | 心跳 Pong | `{ ts }` |
| S→C | `queue_status` | 排队状态 | `{ position, total, estimatedWait }` |
| S→C | `agent_info` / `phase_change` | 客服信息/阶段变更 | AgentInfo / phase 等 |
| S→C | `sync_response` / `history_response` | 同步/历史结果 | `{ messages, hasMore? }` |
| S→C | `session_switched` | 会话切换（多会话） | `{ conversationId, messages, agentInfo, ... }` |
| S→C | `presence_update` / `read_receipt` / `reaction_update` | 在线/已读/反应 | 对应结构 |
| S→C | `message_edit` / `message_recall` | 编辑/撤回回执 | `{ messageId, content? }` |
| S→C | `kicked` | 被踢下线 | `{ reason }` |
| S→C | `error` | 错误 | `{ code, message }` |

### 3.3 序列化格式：JSON 与 Protobuf

- **JSON**（默认）：可读性好、易调试、兼容性佳；体积与解析开销相对较大。  
- **Protobuf**：高 QPS 场景下序列化更快、体积更小（约 10 倍速度、约 60% 体积），需约定 schema。

**协商方式**：

- 连接时通过 URL 参数 `format=json` 或 `format=protobuf` 协商（默认 `json`）。  
- 客户端在 Header 提供「JSON / Protobuf」下拉，选择后写入 Store 并持久化，下次连接生效。

**实现要点**：

- `serializer.ts`：`encodeFrame(frame, format)`、`decodeFrame(data, format)`；Protobuf 使用内联 proto 与 `protobufjs` 解析。  
- 高流量 payload（如 `message`、`message_ack`、`ping`/`pong`、`queue_status`）在 Protobuf 下有专用 message 定义；其余仍以 JSON 字节放入 `payload` 字段。

### 3.4 大消息分片传输（避免阻塞与心跳超时）

**问题**：单条 WebSocket 消息过大时，会长时间占用连接，导致心跳超时或阻塞其他帧。

**方案**：应用层分片——单条「逻辑帧」编码后若超过 **64KB**，拆成多条 WebSocket 消息发送；接收方按「分片 ID + 总片数」重组后再解码为一帧。

**常量**：

- `CHUNK_SIZE = 64 * 1024`（64KB），单分片上限。

**发送流程**：

1. 按当前格式（JSON 或 Protobuf）编码整帧，得到 `encoded`（string 或 ArrayBuffer）。  
2. 若 `encoded` 字节长度 ≤ 64KB：直接发送一条 WebSocket 消息。  
3. 若 > 64KB：  
   - 生成 `messageId`（如 UUID 或 `frag-${timestamp}-${random}`）；  
   - 使用 `splitIntoChunks(encoded)` 得到 `chunks` 与 `totalChunks`；  
   - 先发**一条文本消息**：`createFragMeta(messageId, totalChunks, format)`（JSON 字符串）；  
   - 再按顺序发 **N 条二进制消息**，每条为一块 ≤64KB 的 chunk。

**接收流程**：

1. **文本消息**：`JSON.parse`；若 `isFragMeta(parsed)`，则进入「分片接收」状态，记录 `messageId`、`totalChunks`、`format`，并清空该 messageId 的 chunk 缓冲区。  
2. **二进制消息**：  
   - 若当前处于该 messageId 的分片接收状态：将本条二进制按顺序 push 到 chunks；当 `chunks.length === totalChunks` 时，`reassembleChunks(chunks)` 得到完整 buffer，再按 `format` 调用 `decodeFrame(reassembled, format)` 得到 Frame，交给业务逻辑，并清空该 messageId 状态。  
   - 若未在分片状态：将本条二进制视为**完整一帧**，按当前 format 解码并处理。  
3. 连接断开时清空分片状态（客户端 `disconnect()`、服务端连接关闭时清理）。

**分片元数据格式**（仅文本，首条发送）：

```json
{
  "type": "frag_meta",
  "payload": {
    "messageId": "frag-1739xxx-abc12",
    "totalChunks": 3,
    "format": "json"
  }
}
```

**效果**：单条 WebSocket 消息均 ≤64KB，不会因大消息长时间占用连接，心跳可正常进行；接收端重组后对上层仍是一次完整 Frame。

## 4. 高 QPS 与大消息处理

### 4.1 消息批处理（MessageQueue）

- **窗口**：如 100ms 内的多条发送合并为一批。  
- **批量大小**：如每批最多 20 条，凑满或超时即发送。  
- **收益**：减少 React 渲染次数（从 N 次到 ceil(N/20)）、降低 WebSocket 写次数。  

### 4.2 服务端限流

- **维度**：按用户（userId）限流。  
- **规则**：如每用户每秒最多 200 条消息（`RATE_LIMIT_MSGS_PER_SEC`），滑动窗口统计。  
- **超限**：返回 `error` 帧，`code: "rate_limit"`，不落库。  

### 4.3 序列号与去重

- 发送端：每帧 `seq` 单调递增；重试时复用同一 `seq`，实现幂等。  
- 接收端：可根据 `seq` 去重、检测空洞并触发 sync/history。  
- MessageQueue 内部对消息 ID 做去重，避免重复展示。  

### 4.4 虚拟列表（长列表）

- 消息数 > 1000 时，采用虚拟滚动（如 react-virtuoso）只渲染可视区域，保证 DOM 数量可控。  
- 历史页 `/history` 已做虚拟列表与性能验证。  

### 4.5 大消息与分片（小结）

- 编码后 > 64KB 的帧自动分片发送；接收端按 `frag_meta` + 顺序二进制块重组。  
- 单分片传输不阻塞连接，避免大消息导致心跳超时。

## 5. 状态管理与数据流

### 5.1 会话阶段机（ConversationPhase）

```
BOT → (用户点击转人工) → QUEUING → (排队到 0) → AGENT → (结束) → CLOSED
```

- **BOT**：FAQ、关键词回复、可请求转人工。  
- **QUEUING**：排队中，展示 position/total、预估等待时间。  
- **AGENT**：已分配客服，完整聊天能力。  
- **CLOSED**：会话结束。  

### 5.2 Store 与持久化

- **chatStore**：认证、连接态、会话、消息列表、输入态、UI 状态、**序列化格式**等。  
- **持久化**：Zustand `persist` + IndexedDB（`chatPersistStorage`），持久化字段包括 `messages`、`conversationId`、`format`。  
- **离线恢复**：auth_ok 若服务端下发的消息为空，则通过 `getPersistedMessages(conversationId)` 从 IndexedDB 读取本地消息展示。  

### 5.3 事件流（SDKEvent → Store → UI）

- IMClient 基于 EventEmitter 派发事件（如 `MESSAGE_RECEIVED`、`MESSAGE_BATCH_RECEIVED`、`MESSAGE_STATUS_UPDATE`、`PHASE_CHANGED` 等）。  
- Store 订阅这些事件，在回调中 `set(...)` 更新状态（消息列表、阶段、连接态等）。  
- UI 通过 `useChatStore` 订阅状态，状态变化触发重渲染。  
- 引用隔离：Store 中 push 消息时使用 `{ ...message }` 拷贝，避免与 SDK 内部引用共享，配合 Immer 的 freeze。

## 6. 安全与认证

### 6.1 认证方式

- **WebSocket**：连接 URL 携带 `?token=xxx`；服务端在 `handleConnection` 中校验 token，无效则关闭连接并发送 `error` 帧。  
- **REST**：上传、搜索等接口通过 `Authorization: Bearer <token>` 校验。  
- **Token 来源**：SIWE 钱包登录或 Demo 访客登录，由后端签发 JWT。  

### 6.2 多端与踢人

- 同一 userId 多端连接时，可配置「踢掉其他端」：新连接建立后向旧连接发送 `kicked` 并关闭旧连接。  
- 连接按 `userId → connId` 维护（Map<userId, Map<connId, ConnEntry>>）。  

### 6.3 限流（见 4.2）

- 服务端按用户限流，防止单用户压垮通道。

## 7. 部署与扩展

### 7.1 当前部署形态

- **前端**：Next.js，默认端口 3000。  
- **后端**：Express + ws，默认端口 3001；静态文件、REST、WebSocket 同进程。  
- **数据**：better-sqlite3 单机持久化。  

### 7.2 扩展方向（生产化）

| 方向 | 建议 |
|------|------|
| **水平扩展** | WebSocket 与 REST 分离；多实例 + Redis Pub/Sub 做消息广播与会话绑定 |
| **持久化** | 消息、会话迁至 PostgreSQL/MongoDB；分页与游标拉取 |
| **文件** | 上传走 S3/OSS，CDN 分发；大文件可结合分片上传 |
| **虚拟列表** | 生产环境消息列表统一使用 react-virtuoso 等 |
| **推送** | Service Worker + 推送 API，离线提醒 |
| **安全** | 敏感内容端到端加密、审计日志、合规存储 |
