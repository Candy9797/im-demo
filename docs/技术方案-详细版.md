## 1. 项目概述与目标

### 1.1 项目定位

面向 Web3 交易所的**客服即时通讯（IM）Demo**，参考 Binance 客服体系，实现：

- **Bot 阶段**：FAQ、关键词回复、一键转人工  
- **人工阶段**：排队、分配客服、完整聊天（文本/图片/PDF/表情/已读/反应/编辑/撤回）  
- **架构能力**：WebSocket 长连接、双格式（JSON/Protobuf）、大消息分片、批处理与限流、序列号与去重  

### 1.2 核心目标与对应手段

| 目标 | 手段 |
|------|------|
| 实时性 | WebSocket 长连接、心跳 Ping/Pong、断线指数退避重连、乐观更新 |
| 可扩展 | 帧级协议、seq、JSON/Protobuf 可选、>64KB 自动分片 |
| 高 QPS | MessageQueue 批处理/去重/重试、服务端按用户限流、虚拟列表 |
| 可维护 | SDK 与 UI 解耦、EventEmitter、Zustand、类型完备 |

### 1.3 技术栈与仓库结构

- **前端**：Next.js 16（App Router）、React 19、TypeScript、Zustand、IndexedDB（持久化）  
- **后端**：Node.js、Express、ws、better-sqlite3  
- **SDK**：自研（`src/sdk/`），无第三方 IM 库；序列化与分片在 `serializer.ts`  

**关键目录**：

- `src/app/`：页面与路由  
- `src/components/`：聊天 UI 组件  
- `src/store/`：chatStore、持久化  
- `src/sdk/`：IMClient、WebSocketManager、MessageQueue、serializer、types  
- `server/`：Express + ws、ws-handler、db、auth、bot、upload  
- `proto/`：im.proto（Protobuf 定义，与 serializer 内联一致）

## 3. 协议设计

### 3.1 帧结构（Frame）

所有 C2S / S2C 消息均为**一帧**，结构固定：

```typescript
interface Frame {
  type: FrameType;   // 字符串枚举，如 "send_message", "message", "auth_ok"
  seq: number;      // 单调递增，发端维护；用于顺序、去重、空洞检测、幂等重试
  timestamp: number; // 毫秒时间戳
  payload: unknown;  // 由 type 决定结构，见下表
}
```

- **seq**：发送方每发一帧自增；重试时复用同一 seq，接收方可去重。  
- **payload**：类型与 `type` 一一对应，见下节。  

### 3.2 帧类型与 payload 结构（完整）

**C2S（Client → Server）**

| type | 说明 | payload 结构 |
|------|------|--------------|
| `auth` | 认证（实际以 URL ?token= 为主） | 可选空或补充信息 |
| `send_message` | 发送消息 | `Message` 或 `Message[]` |
| `typing` | 正在输入 | `{ senderType?: string }` |
| `ack` | 消息确认 | 可选 |
| `ping` | 心跳 Ping | `{ ts: number }` |
| `request_agent` | 请求转人工 | `{ conversationId, userId }` |
| `sync` | 同步（拉取 afterSeqId 之后） | `{ afterSeqId?: number }` |
| `load_history` | 拉取历史（beforeSeqId 之前） | `{ beforeSeqId?: number }` |
| `mark_read` | 标记已读 | `{ messageIds: string[] }` |
| `add_reaction` | 添加表情反应 | `{ messageId: string, emoji: string }` |
| `remove_reaction` | 移除表情反应 | `{ messageId: string, emoji: string }` |
| `edit_message` | 编辑消息 | `{ messageId: string, content: string }` |
| `recall_message` | 撤回消息 | `{ messageId: string }` |
| `simulate_push` | 测试压测 | `{ count?: number }` |

**S2C（Server → Client）**

| type | 说明 | payload 结构 |
|------|------|--------------|
| `auth_ok` | 认证成功 | `{ conversationId, sessionType, phase, messages: Message[], hasMore }` |
| `message` | 收到消息 | `Message` 或 `Message[]` |
| `message_ack` | 消息送达确认 | `{ clientMsgId, serverMsgId, seqId }[]` |
| `typing_start` / `typing_stop` | 输入状态 | `{ senderType? }` |
| `pong` | 心跳 Pong | `{ ts: number }` |
| `queue_status` | 排队状态 | `{ position, total, estimatedWait }` |
| `agent_info` | 分配客服信息 | `AgentInfo` |
| `phase_change` | 阶段变更 | `{ phase, queuePosition?, queueTotal?, estimatedWait? }` |
| `sync_response` | 同步结果 | `{ messages: Message[] }` |
| `history_response` | 历史结果 | `{ messages: Message[], hasMore }` |
| `session_switched` | 会话切换 | `{ conversationId, sessionType, phase, agentInfo?, messages }` |
| `presence_update` | 在线列表 | `{ online: string[], timestamp }` |
| `read_receipt` | 已读回执 | `{ messageIds, readBy }` |
| `reaction_update` | 反应更新 | `{ messageId, clientMsgId?, reactions, message? }` |
| `message_edit` | 编辑回执 | `{ messageId, content }` |
| `message_recall` | 撤回回执 | `{ messageId }` |
| `kicked` | 被踢下线 | `{ reason }` |
| `error` | 错误 | `{ code, message?, messageId? }` |

### 3.3 Message 与 AgentInfo 结构

```typescript
interface Message {
  id: string;              // 客户端生成或服务端 serverMsgId
  conversationId: string;
  content: string;         // 文本或 URL（图片/文件）
  type: MessageType;       // "text" | "image" | "pdf" | "system" | ...
  status: MessageStatus;   // "sending" | "sent" | "delivered" | "read" | "failed"
  senderType: SenderType;  // "user" | "bot" | "agent" | "system"
  senderId: string;
  senderName: string;
  senderAvatar?: string;
  timestamp: number;
  metadata?: MessageMetadata; // reactions, readBy, quote, ...
  seqId?: number;          // 服务端序号，排序与 sync/history
}

interface AgentInfo {
  id: string;
  name: string;
  code: string;
  avatar?: string;
  department?: string;
}
```

### 3.4 序列化格式：JSON 与 Protobuf

- **协商**：连接 URL 增加 `format=json` 或 `format=protobuf`（默认不传为 json）；客户端 Header 下拉选择并持久化到 Store（format 参与 partialize）。  
- **JSON**：整帧 `JSON.stringify(frame)`；文本 WebSocket 消息。  
- **Protobuf**：  
  - 帧封装为 `im.Frame`：`type`（枚举 int）、`seq`、`timestamp`、`payload`（bytes）。  
  - 高流量 payload 使用专用 message：如 `IMMessage`、`MessageAckPayload`、`HeartbeatPayload`、`QueueStatusPayload` 等；其余 payload 以 JSON 序列化后写入 bytes。  
  - 二进制 WebSocket 消息。  
- **自动识别**：接收时若未显式指定 format，可根据首字节非 `{`/`[` 推断为二进制并按 Protobuf 解码（见 `serializer.ts` `looksLikeProtobuf`）。  

### 3.5 分片控制帧（frag_meta）

大消息分片时，**首条**为文本消息，且**不是**业务 Frame，而是分片元数据：

```json
{
  "type": "frag_meta",
  "payload": {
    "messageId": "frag-1739xxx-abc12",
    "totalChunks": 3,
    "format": "json"
  }
}
```

- `messageId`：本次分片组的唯一 ID，用于接收端重组。  
- `totalChunks`：紧随其后的二进制消息条数（即分片数）。  
- `format`：重组后整帧的序列化格式，用于 `decodeFrame(reassembled, format)`。  

之后按顺序发送 `totalChunks` 条**二进制** WebSocket 消息，每条为 ≤64KB 的 chunk；接收端按顺序拼接后解码为一帧。

## 4. 连接与心跳与重连

### 4.1 WebSocket 连接状态机（WebSocketManager）

```
DISCONNECTED ──connect()──► CONNECTING
                                │
                    onopen      ▼
                    ─────────► CONNECTED ──disconnect()──► DISCONNECTED
                                │
                    onclose     │
                    ─────────► RECONNECTING ──重试成功──► CONNECTED
                                │
                    达到最大重试 ──► DISCONNECTED
```

- **CONNECTING**：首次连接或重连中，不重复发起连接。  
- **CONNECTED**：可 send、收 onmessage、启动心跳定时器。  
- **RECONNECTING**：onclose 后若未达最大次数则进入，按指数退避 scheduleReconnect。  
- **disconnect()**：主动断开时清除心跳、重连定时器、分片状态，并置为 DISCONNECTED。  

### 4.2 URL 与参数

- 基础：`ws://host:3001/ws`  
- 参数：`token`（必填，认证）、`fresh=1`（可选，新建会话）、`format=protobuf`（可选，默认 json）。  
- 示例：`ws://host:3001/ws?token=eyJ...&fresh=1&format=protobuf`  

### 4.3 心跳

- **间隔**：`heartbeatInterval`，默认 30000ms。  
- **发送**：定时器内若状态为 CONNECTED 则 `send(HEARTBEAT_PING, { ts: Date.now() })`。  
- **接收**：服务端回复 `HEARTBEAT_PONG`，客户端仅忽略或用于存活判断。  
- **作用**：保持连接活跃、及早发现断线；大消息分片后单条 ≤64KB，避免写阻塞导致心跳发不出去。  

### 4.4 重连策略

- **触发**：WebSocket `onclose`（含网络异常、服务端关闭）。  
- **上限**：`reconnectAttempts`，默认 5。  
- **间隔**：指数退避 + 抖动，`delay = min(baseInterval * 2^reconnectCount + random(0,1000), 30000)` ms。  
- **行为**：进入 RECONNECTING，emit RECONNECTING，到期后再次 `connect()`；成功后 `reconnectCount` 置 0。  
- **与 MessageQueue**：断线时 IMClient 调用 `messageQueue.pause()` 与 `rollbackPendingAck()`；重连成功后 `resume()`，未确认消息会重新从 outgoing 发出。

## 5. 消息队列与高 QPS

### 5.1 MessageQueue 角色

- **出站**：`enqueueOutgoing(message)` → 定时 `flush` 取一批 → `onFlushOutgoing(messages)`（实际 `wsManager.send(SEND_MESSAGE, batch)`）→ 发送成功后移入 `pendingAck`，启动 ACK 超时定时器。  
- **入站**：收到消息后 IMClient 调用 `enqueueIncoming(message)`（带去重）→ 定时 `flush` 取一批 → `onFlushIncoming(batch)`（派发 MESSAGE_BATCH_RECEIVED）。  
- **顺序**：每次 flush 先 `flushIncoming()` 再 `flushOutgoing()`，优先展示收到的新消息。  

### 5.2 配置项（QueueConfig）

| 字段 | 默认 | 说明 |
|------|------|------|
| maxSize | 1000 | 出队最大长度，超限丢弃最旧非 sending 消息；**IMClient 传入 2000** |
| batchSize | 20 | 每批条数；**IMClient 传入 300** |
| flushInterval | 100 | 批量 flush 间隔（ms）；**IMClient 传入 50** |
| retryAttempts | 3 | 发送失败/ACK 超时最大重试次数 |
| retryDelay | 1000 | 重试间隔基数（指数退避） |
| deduplicationWindow | 5000 | 入站去重窗口（ms） |
| ackTimeoutMs | 10000 | ACK 超时（ms），超时触发重发或失败 |

### 5.3 待确认队列（pendingAck）与 ACK 超时

- **发送成功**：batch 中每条消息放入 `pendingAck`，key 为 `clientMsgId`，并设 `setTimeout(handleAckTimeout, ackTimeoutMs)`。  
- **收到 message_ack**：IMClient 根据 payload 中的 `clientMsgId` 调用 `messageQueue.onAck(clientMsgId)`，移除 pending 并清除定时器，更新消息状态为 sent。  
- **超时**：`handleAckTimeout` 中若重试次数未用尽则重新 `unshift` 回 `outgoing`，否则 `markSendFailed` 并触发 `onMessageSendFailed`（Store 收 MESSAGE_SEND_FAILED）。  
- **断线**：`rollbackPendingAck()` 将 pendingAck 中未确认消息回滚到 `outgoing`（或标记失败），重连后由 flush 再次发出。  

### 5.4 入站去重

- **seenIds**：Map<messageId, timestamp>，仅保留在 `deduplicationWindow` 内的条目。  
- **enqueueIncoming** 时若 `seenIds.has(message.id)` 则丢弃；否则 `seenIds.set` 并 push 到 `incoming`。  
- **cleanupDedup**：在入队时顺带清理过期 id，控制 Map 大小。  

### 5.5 服务端限流

- **维度**：按 `userId`。  
- **规则**：滑动窗口 1 秒，每用户最多 `RATE_LIMIT_MSGS_PER_SEC`（200）条；`checkRateLimit(userId)` 在每次 SEND_MESSAGE 处理时调用，超限则对该用户回复 `error`（code: "rate_limit"），该条不落库。  
- **实现**：`rateLimitMap: Map<userId, timestamps[]>`，每次发送前过滤掉 1 秒外的 ts 再判断 length。

## 6. 大消息分片

### 6.1 阈值与常量

- **CHUNK_SIZE** = 64 * 1024（字节）。  
- 编码后整帧长度 **> 64KB** 时走分片；否则单条 WebSocket 消息发送。  

### 6.2 发送流程（发送端）

1. 构造业务 Frame，调用 `encodeFrame(frame, format)` 得到 `encoded`（string 或 ArrayBuffer）。  
2. 计算字节长度：  
   - 若为 string：`new TextEncoder().encode(encoded).length`（客户端）或 `Buffer.byteLength(encoded, "utf8")`（服务端）。  
   - 若为 ArrayBuffer：`encoded.byteLength`。  
3. 若 `size <= CHUNK_SIZE`：`ws.send(encoded)`，结束。  
4. 若 `size > CHUNK_SIZE`：  
   - 生成 `messageId`（如 `crypto.randomUUID()` 或 `frag-${Date.now()}-${random}`）。  
   - `splitIntoChunks(encoded)` → `{ chunks, totalChunks }`（每块 ≤64KB）。  
   - `ws.send(createFragMeta(messageId, totalChunks, format))`（文本）。  
   - 对每个 chunk：`ws.send(chunk.buffer.slice(chunk.byteOffset, chunk.byteOffset + chunk.byteLength))`（或服务端 `Buffer.from(chunk)`）。  

### 6.3 接收流程（接收端）

1. **文本消息**：  
   - `JSON.parse(data)`。  
   - 若 `isFragMeta(parsed)`：设置/更新分片状态（`messageId`、`totalChunks`、`format`，`chunks = []`），return。  
   - 否则按普通帧：`decodeFrame(data, format)` → `handleFrame(frame)`。  

2. **二进制消息**：  
   - 若当前存在该连接的分片状态（如 `fragmentState`）：  
     - 将当前二进制 push 到 `chunks`。  
     - 若 `chunks.length >= totalChunks`：`reassembleChunks(chunks)` 得到完整 buffer，`decodeFrame(reassembled, format)` 得到 Frame，`handleFrame(frame)`，然后清空分片状态。  
   - 若无分片状态：将本条二进制视为完整一帧，`decodeFrame(data, format)` → `handleFrame(frame)`。  

3. **连接关闭**：客户端在 `disconnect()` 中清空 `fragmentState`；服务端在连接 close 时清空该 ConnEntry 的 `fragmentState`。  

### 6.4 边界与约束

- **顺序**：分片按发送顺序接收并 push，重组时按数组顺序拼接，**不支持乱序**（若需可扩展为按 chunk index 排序后再拼接）。  
- **多组并发**：当前实现为单连接一个分片状态（一个 messageId）；若连续收到多个 frag_meta，后者会覆盖前者，前一组未收齐的 chunk 会污染后一组，因此**同一连接上不宜并发发送多组大消息**（通常业务上单帧大消息已少见）。  
- **断线**：收齐前断线则分片状态被清空，该大帧需依赖业务层重试（如重新请求 sync/history）。  

### 6.5 代码位置

- 常量与工具：`src/sdk/serializer.ts`（CHUNK_SIZE、createFragMeta、isFragMeta、splitIntoChunks、reassembleChunks）。  
- 客户端发送与接收：`src/sdk/WebSocketManager.ts`（send 内 size 判断与分片发送，handleMessage 内 frag_meta 与二进制重组）。  
- 服务端发送与接收：`server/ws-handler.ts`（send 内分片、ConnEntry.fragmentState、on("message") 内解析与重组）。

## 7. 服务端处理流程

### 7.1 连接建立（handleConnection）

1. 从 URL 取 `token`、`fresh`、`format`（默认 json）。  
2. 无 token → send error "auth_required" 并 close。  
3. `verifyToken(token)` 失败 → send error "invalid_token" 并 close。  
4. 根据 fresh 创建或复用 Bot 会话（convId）。  
5. 若 kickOthers：关闭该 userId 下其它连接并 send KICKED。  
6. 构造 ConnEntry（ws、userId、address、convId、connId、outSeq、format、fragmentState: null），写入 connsByUser / wsToConn。  
7. 拉取初始消息（如最近 INITIAL_LOAD_LIMIT 条），send AUTH_OK（conversationId、messages、hasMore、phase 等）。  
8. broadcastPresence()。  
9. 注册 ws.on("message", ...) 与 ws.on("close", ...)。  

### 7.2 消息处理（handleFrame）概要

- **ping**：回复 pong。  
- **load_history**：按 beforeSeqId 分页查库，回复 history_response。  
- **mark_read**：更新消息状态与 metadata.readBy，回复 read_receipt。  
- **add_reaction / remove_reaction**：更新 metadata.reactions，回复 reaction_update。  
- **edit_message**：校验归属与会话，更新 content，向会话内广播 message_edit。  
- **recall_message**：校验归属、2 分钟内可撤回，更新内容与 metadata，广播 message_recall。  
- **send_message**：限流检查；每条落库、生成 serverMsgId/seqId；Bot 阶段可触发关键词回复或转排队；Agent 阶段可插入模拟客服回复；批量 ACK 通过 message_ack 下发。  
- **request_agent**：创建 Agent 会话、更新排队状态、定时模拟排队递减、assignAgent 后发送 session_switched。  
- **sync**：按 afterSeqId 查库，回复 sync_response。  
- **simulate_push**：测试用，批量插入 Bot 消息并推送。  

### 7.3 发送出口

- **send(ws, type, payload, format)**：按 entry 的 format 编码；若编码后 > CHUNK_SIZE 则分片发送，否则单条发送。  
- **sendToUser(userId, type, payload)**：向该 userId 下所有连接发送（每连接用自身 format）。  
- **sendToConversation(convId, type, payload)**：向该会话关联的 user 与 agent 分别 sendToUser。

## 8. 状态管理与持久化

### 8.1 chatStore 状态分组

- **认证**：auth（token、userId、address）、authError、authConnecting。  
- **连接**：connectionState、client（IMClient 实例，不持久化）。  
- **会话**：conversationId、phase、agentInfo、queue、messages、faqItems。  
- **辅助**：typing、hasMoreHistory、loadingHistory、onlineUsers。  
- **UI**：isMinimized、isExpanded、isOpen、showWalletModal、wantFreshStart、searchResults、quoteTarget、scrollToInputRequest、toast、**format**。  

### 8.2 持久化（persist）

- **storage**：createJSONStorage(() => chatPersistStorage)，底层为 IndexedDB（防抖写入）。  
- **partialize**：仅持久化 `messages`、`conversationId`、`format`。  
- **name**：CHAT_PERSIST_NAME（"web3-im-chat"）。  
- **原因**：client、auth、connectionState 等为运行时或敏感信息，刷新后需重新连接与登录；messages/conversationId/format 用于恢复最近对话与格式偏好。  

### 8.3 离线恢复（getPersistedMessages）

- **时机**：auth_ok 时若服务端下发的 messages 为空（如新会话或服务端未存历史），IMClient 会调用 config.getPersistedMessages(conversationId)。  
- **实现**：Store 在 createIMClient 时传入闭包，内部调用 getPersistedChatState() 从 IndexedDB 读取；若 conversationId 匹配且有 messages 则返回，否则返回空数组。  
- **作用**：避免 rehydration 未完成时界面短暂空白，优先展示本地已有消息。  

### 8.4 会话阶段与事件映射

- BOT → QUEUING：用户 request_agent，收到 phase_change(queue_status) 等。  
- QUEUING → AGENT：收到 session_switched（含 agentInfo、messages）。  
- Store 订阅 PHASE_CHANGED、QUEUE_UPDATE、AGENT_ASSIGNED 等，更新 phase、queue、agentInfo。

## 9. 安全、限流与多端

### 9.1 认证

- **WebSocket**：URL 必带有效 token；服务端在 handleConnection 首行校验，失败即关闭。  
- **REST**：上传、搜索等需 `Authorization: Bearer <token>`，中间件校验。  
- **Token**：JWT，由 auth 模块签发（SIWE 或 Demo 访客），含 userId、address 等。  

### 9.2 多端与踢人

- **connsByUser**：Map<userId, Map<connId, ConnEntry>>，同一用户可有多个连接（多设备）。  
- **kickOthers**：新连接建立时若为 true，则关闭该 userId 下除当前 ws 外的所有连接，并先发 KICKED 再 close。  
- **KICKED**：客户端收到后通常清空 auth、断开并提示“已在其他设备登录”。  

### 9.3 限流（见 5.5）

- 仅对 SEND_MESSAGE 做按用户限流；其它帧类型不限。  
- 调试接口：GET /api/rate-limit-state、/api/rate-limit-config。
