# 消息「秒达」监控与保障

本文档描述 IM 系统中实现消息秒级到达的监控方案与保障措施。

---

## 一、目标与定义

### 1.1 秒达定义

- **发送端**：用户点击发送到本地展示（乐观更新）— 期望 < 100ms
- **端到端**：用户 A 发送 → 用户 B 收到 — 期望 < 1s（秒达）
- **ACK 确认**：发送到收到服务端 message_ack — 期望 < 500ms

### 1.2 关键路径

```
发送端：用户输入 → sendMessage → 乐观更新 → MessageQueue 出队 → ws.send
    ↓
网络：WebSocket 帧传输
    ↓
服务端：handleFrame → 入库 → send(MESSAGE/MESSAGE_ACK)
    ↓
接收端：ws.onmessage → MessageQueue 入队 → flush → UI 更新
```

## 二、监控指标

### 2.1 客户端埋点

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| `send_to_ack_ms` | 发送到 ACK 耗时 | `ackTimestamp - sendTimestamp` |
| `receive_to_render_ms` | 收包到渲染耗时 | `renderTimestamp - receiveTimestamp` |
| `frame_in_latency_ms` | 服务端推送到客户端处理 | 在 frame payload 中带服务端 `ts`，客户端计算差值 |

### 2.2 服务端埋点

| 指标 | 说明 | 计算方式 |
|------|------|----------|
| `recv_to_db_ms` | 收到帧到入库完成 | `dbDoneTs - recvTs` |
| `recv_to_send_ms` | 收到帧到发出 MESSAGE/ACK | `sendTs - recvTs` |
| `queue_depth` | 待处理帧队列长度 | 可选，用于发现积压 |

### 2.3 实现示例

**客户端**（在 IMClient / WebSocketManager 中）：
```typescript
// 发送时
const sendTs = Date.now();
this.wsManager.send(FrameType.SEND_MESSAGE, { ...msg, _sendTs: sendTs });

// ACK 时
this.wsManager.on("message_ack", (payload) => {
  const ackTs = Date.now();
  const sendTs = (payload as { _sendTs?: number })._sendTs;
  if (sendTs) {
    const latency = ackTs - sendTs;
    reportMetric("send_to_ack_ms", latency);
    if (latency > 1000) reportSlowMessage(latency);
  }
});
```

**服务端**（在 ws-handler 中）：
```typescript
const recvTs = Date.now();
// ... 处理 SEND_MESSAGE ...
const sendTs = Date.now();
send(ws, FrameType.MESSAGE_ACK, { ...payload, _serverLatency: sendTs - recvTs });
```

## 三、保障措施

### 3.1 客户端

| 措施 | 说明 | 当前实现 / 建议 |
|------|------|------------------|
| 乐观更新 | 发送即展示，不等待 ACK | 已实现 |
| 出队间隔 | MessageQueue flushInterval | 50–100ms，不宜过大 |
| 批处理大小 | batchSize 影响单次发送条数 | 建议 ≤ 20 |
| 避免阻塞主线程 | IndexedDB 异步、防抖 | 已实现 |
| WebSocket 长连 | 避免频繁建连 | 已实现 |

### 3.2 服务端

| 措施 | 说明 | 建议 |
|------|------|------|
| 快速路径 | SEND_MESSAGE → 入库 → 回 ACK/MESSAGE，无额外 RPC | 当前实现已满足 |
| 批量写入 | insertMessages 事务，减少 DB 往返 | 已实现 |
| 限流 | 避免单用户突发拖垮服务 | RATE_LIMIT 20/秒，可按需调整 |
| 连接与实例 | 单机内存处理，无跨进程转发 | 多实例时需消息路由与负载均衡 |

### 3.3 网络

| 措施 | 说明 |
|------|------|
| 同域/同区部署 | 客户端与 IM 服务同地域，降低 RTT |
| 心跳保活 | 避免中间件因空闲断开，当前 30s 心跳 |
| 重连策略 | 指数退避，减少断线期间消息堆积 |
| CDN/边缘 | 静态资源走 CDN，IM 连接直连网关 |

## 四、告警与 SLO

### 4.1 建议 SLO

| 指标 | P99 目标 | 告警阈值 |
|------|----------|----------|
| send_to_ack_ms | < 500ms | > 2000ms |
| recv_to_send_ms（服务端） | < 100ms | > 500ms |
| 连接失败率 | < 0.1% | > 1% |
| 消息丢失率 | 0 | 任一类丢消息即告警 |

### 4.2 告警规则示例

- `send_to_ack_ms` 连续 5 分钟 P99 > 2s → 告警
- 服务端 `recv_to_send_ms` P99 > 500ms → 告警
- WebSocket 断线率 > 1% → 告警
- 消息队列积压（如 outgoing 持续 > 100）→ 告警

---

## 五、压测与调优

### 5.1 压测场景

- 单用户高频发送（如 10 条/秒）
- 多用户并发（如 100 用户同时在线收发）
- 突发流量（短时间内大量连接与消息）

### 5.2 可调参数

| 模块 | 参数 | 默认 | 说明 |
|------|------|------|------|
| MessageQueue | flushInterval | 100 | 减小可降低 ACK 延迟，增大可减少帧数 |
| MessageQueue | batchSize | 20 | 单次发送条数 |
| IndexedDBStore | WRITE_DEBOUNCE_MS | 80 | 写入防抖，影响持久化延迟 |
| WebSocketManager | heartbeatInterval | 30000 | 心跳间隔 |
| ws-handler | RATE_LIMIT_MSGS_PER_SEC | 20 | 单用户限流 |

---

## 六、小结

- **监控**：客户端 `send_to_ack_ms`、服务端 `recv_to_send_ms` 为核心指标，配合日志与指标上报
- **保障**：乐观更新、合理批处理、快速服务端路径、限流与部署优化
- **运维**：定义 SLO、告警规则，定期压测与参数调优
