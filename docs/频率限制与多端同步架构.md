# 频率限制与多端同步架构

本文档描述 IM 系统的**单用户 QPS 频率限制**实现方式，以及**多端在线且消息同步**的架构设计要点。

---

## 一、频率限制（防刷屏）

### 1.1 目标

- 限制单用户发送消息的 QPS，防止恶意刷屏
- 保证正常用户不受影响，恶意用户被限流

### 1.2 当前实现

**位置**：`server/ws-handler.ts`

**算法**：滑动窗口计数

```typescript
const RATE_LIMIT_MSGS_PER_SEC = 20;   // 每秒最多 20 条
const RATE_WINDOW_MS = 1000;          // 窗口 1 秒

function checkRateLimit(userId: string): boolean {
  const now = Date.now();
  let timestamps = rateLimitMap.get(userId);
  if (!timestamps) {
    rateLimitMap.set(userId, [now]);
    return true;
  }
  timestamps = timestamps.filter((t) => now - t < RATE_WINDOW_MS);
  if (timestamps.length >= RATE_LIMIT_MSGS_PER_SEC) return false;
  timestamps.push(now);
  rateLimitMap.set(userId, timestamps);
  return true;
}
```

**流程**：
1. 收到 `SEND_MESSAGE` 帧时先调用 `checkRateLimit(userId)`
2. 通过则继续处理；不通过则发送 `ERROR { code: "rate_limit", message: "Too many messages, please slow down" }`，不入库

### 1.3 实现要点

| 要点 | 说明 |
|------|------|
| 按 userId 限流 | 同一用户多端共享同一限制，避免多端绕过 |
| 滑动窗口 | 保留 1 秒内的发送时间戳，超窗口的剔除，避免突发后在下一秒立刻恢复 |
| 内存存储 | `rateLimitMap` 单机内存，多实例需共享（见下）|

### 1.4 可选实现方式

| 方案 | 优点 | 缺点 | 适用 |
|------|------|------|------|
| 滑动窗口计数 | 实现简单，精度较高 | 需存储时间戳列表 | 单机 |
| 令牌桶 | 允许短时突发，平滑限流 | 实现稍复杂 | 单机/分布式 |
| Redis + Lua | 支持多实例共享 | 依赖 Redis，有网络开销 | 分布式 |
| Leaky Bucket | 严格平滑输出 | 实现复杂 | 特定场景 |

**令牌桶示例**（单机）：
```typescript
const tokens = new Map<string, { count: number; lastRefill: number }>();
const CAPACITY = 20;
const REFILL_RATE = 20; // tokens per second

function checkRateLimitTokenBucket(userId: string): boolean {
  const now = Date.now() / 1000;
  let bucket = tokens.get(userId);
  if (!bucket) {
    tokens.set(userId, { count: CAPACITY - 1, lastRefill: now });
    return true;
  }
  const elapsed = now - bucket.lastRefill;
  bucket.count = Math.min(CAPACITY, bucket.count + elapsed * REFILL_RATE);
  bucket.lastRefill = now;
  if (bucket.count < 1) return false;
  bucket.count -= 1;
  return true;
}
```

**Redis 滑动窗口**（多实例）：
```lua
-- KEYS[1]: rate_limit:userId
-- ARGV[1]: window_ms (1000)
-- ARGV[2]: max_count (20)
local key = KEYS[1]
local now = tonumber(ARGV[3]) or redis.call('TIME')[1] * 1000
redis.call('ZREMRANGEBYSCORE', key, 0, now - tonumber(ARGV[1]))
local count = redis.call('ZCARD', key)
if count >= tonumber(ARGV[2]) then
  return 0
end
redis.call('ZADD', key, now, now .. ':' .. math.random())
redis.call('PEXPIRE', key, tonumber(ARGV[1]) + 1000)
return 1
```

### 1.5 限流响应

- **服务端**：返回 `ERROR` 帧，客户端应展示「发送过于频繁，请稍后再试」
- **客户端**：可在 UI 层做二次限流（如发送按钮冷却），减轻服务端压力
- **分级限流**：可按用户等级、会话类型设置不同 QPS（如 VIP 用户 50/秒）

### 1.6 参数建议

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| RATE_LIMIT_MSGS_PER_SEC | 20–50 | 正常用户难以达到，恶意刷屏被拦 |
| RATE_WINDOW_MS | 1000 | 1 秒窗口，与 QPS 语义一致 |

## 二、多端在线与消息同步架构

### 2.1 目标

- 同一用户可在多个设备（Web、App、PC）同时在线
- 各端收到的消息一致，且无重复展示
- 已读、输入中等状态在多端间正确同步

### 2.2 设备管理

**原则**：每个登录设备有独立 WebSocket 连接，共享同一个 userId。

**当前实现**（`server/ws-handler.ts`）：

```typescript
// connsByUser: Map<userId, Map<connId, ConnEntry>>
const entry: ConnEntry = { ws, userId, address, convId, connId };
let userConns = connsByUser.get(userId);
if (!userConns) {
  userConns = new Map();
  connsByUser.set(userId, userConns);
}
userConns.set(connId, entry);  // 同一用户可有多条连接
```

**要点**：

| 要点 | 说明 |
|------|------|
| 连接与用户 | 一个 userId 对应多个 connId，每个 connId 对应一个设备/连接 |
| connId | 每次建连生成唯一 ID，用于区分设备 |
| 踢线策略 | 可选：新设备登录时踢掉旧设备（kickOthers），或允许多端并存 |
| 设备元数据 | 可扩展 ConnEntry，增加 deviceType、deviceId、appVersion 等，用于推送策略与排查 |

### 2.3 消息去重

**原则**：客户端按 `msg_id` 去重，避免同一消息在多端、重推、重连等场景下重复渲染。

**实现**（见 `消息不丢不重技术方案.md`）：

- **MessageQueue**：`seenIds` Map，5 秒窗口内按 `message.id` 过滤
- **Store 合并**：`MESSAGE_RECEIVED`、`sync_response`、`history_response` 合并时，用 `ids.has(m.id)` 过滤
- **服务端**：同一消息只入库一次，推送给该用户所有在线连接

**多端场景**：
- 用户 A 在设备 1 发送 → 服务端入库 → 推送给 A 的所有设备（含设备 2、3）
- 设备 2、3 各自收到 MESSAGE 帧，各自按 msg_id 去重，不会重复展示

### 2.4 状态同步：已读以服务端为准

**原则**：已读状态以服务端为准，维护「用户维度的阅读游标」，避免各端不一致。

**设计**：

| 维度 | 说明 |
|------|------|
| 阅读游标 | 每用户每会话维护 `lastReadSeqId` 或 `lastReadMessageId`，表示已读到的位置 |
| 上报 | 任一端 `markAsRead` 时，服务端更新该用户的 lastReadSeqId |
| 下发 | 新设备建连、重连、拉历史时，服务端返回 lastReadSeqId，客户端据此标记已读 |
| 多端一致 | 设备 1 标记已读 → 服务端更新 → 设备 2 收到 READ_RECEIPT 或下次拉取时同步 |

**存储示例**：
```sql
-- 用户会话阅读游标
CREATE TABLE read_cursors (
  user_id TEXT,
  conversation_id TEXT,
  last_read_seq_id INTEGER,
  updated_at INTEGER,
  PRIMARY KEY (user_id, conversation_id)
);
```

**协议扩展**：
- `MARK_READ` payload 可带 `lastReadSeqId`，服务端更新后广播 `READ_RECEIPT` 给该用户所有设备
- `auth_ok` / `sync_response` 可带各会话的 `lastReadSeqId`，供新端初始化

### 2.5 推送策略

**原则**：同一用户的多端同时在线时，推送至所有在线设备，由各端自行合并去重展示。

**流程**：

```
服务端收到/生成新消息
    │
    ▼
确定目标用户（C2C：对方；群组：群成员）
    │
    ▼
对每个目标用户，获取其所有在线连接
    │
    ▼
sendToUser(userId, MESSAGE, payload)  // 向该用户所有 conn 推送
    │
    ▼
各设备独立收包 → MessageQueue 入队 → seenIds 去重 → UI 更新
```

**当前实现**：
```typescript
function sendToUser(userId: string, type: string, payload: unknown) {
  const conns = connsByUser.get(userId);
  if (!conns) return;
  for (const e of conns.values()) {
    send(e.ws, type, payload);
  }
}
```

**扩展**：
- **弱网设备**：可对延迟高、丢包多的连接降级为拉取模式，减少无效推送
- **App 推送**：移动端离线时，通过 APNs/FCM 推送，在线时走 WebSocket

### 2.6 架构示意

```
                    ┌─────────────────┐
                    │     IM 服务端    │
                    │                 │
                    │  connsByUser:   │
                    │  userId ->      │
                    │  [conn1, conn2] │
                    └────────┬────────┘
                             │
           ┌─────────────────┼─────────────────┐
           │                 │                 │
           ▼                 ▼                 ▼
    ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
    │  设备 1     │   │  设备 2     │   │  设备 3     │
    │  (Web)      │   │  (App)      │   │  (PC)       │
    │  userId: A  │   │  userId: A  │   │  userId: A  │
    │  connId: c1 │   │  connId: c2 │   │  connId: c3 │
    └──────┬──────┘   └──────┬──────┘   └──────┬──────┘
           │                 │                 │
           │  seenIds 去重   │  seenIds 去重   │  seenIds 去重
           │  按 msg_id      │  按 msg_id      │  按 msg_id
           ▼                 ▼                 ▼
           └─────────────────┴─────────────────┘
                     消息列表一致、无重复
```

## 三、批量发送（降帧数）

在高 QPS 下，客户端将**多条出站消息合并为一个 WebSocket 帧**发送，减少帧数与网络开销。详见 [批量发送技术方案](./批量发送技术方案.md)。

| 要点 | 说明 |
|------|------|
| 协议 | SEND_MESSAGE payload 支持 `Message \| Message[]` |
| 限流 | 服务端逐条处理，每条独立 checkRateLimit |
| ACK | 每条消息仍单独回 MESSAGE_ACK，客户端逻辑不变 |
