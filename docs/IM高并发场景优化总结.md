# IM 高并发场景优化总结（面试话术）

> 面向面试官：从前端、后端、SDK 三层说明**遇到的问题 → 为什么这样设计 → 好处 → 如何解决高并发**。  
> 实现以 `src/sdk/MessageQueue.ts`、`src/sdk/IMClient.ts`、`src/components/MessageList.tsx`、`server/ws-handler.ts` 等为准。

---

## 高并发/长列表场景下要解决什么问题（总览）

> **说明**：不全是「高并发」带来的——**高并发**主要指高 QPS（每秒很多条）、更新频率高；**长列表**是总条数多（几千、上万）；**重复/乱序**来自多路到达与弱网；**大帧**来自单次 payload 过大。下表放在一起是因为 IM 场景里这些问题常同时出现，方案也一起做。

| 问题 | 现象 | 对应手段 | 与高并发关系 |
|------|------|----------|--------------|
| 更新过频 | 每秒几十条消息，来一条 setState 一次 → 大量 re-render、列表/输入卡顿 | SDK 入站批处理、前端虚拟列表 | **高并发**：消息速率高时问题被放大 |
| 长列表 DOM 爆炸 | 几千、上万条全量渲染 → 首屏慢、滚动卡、内存高 | 前端虚拟列表（Virtuoso） | **长列表/数据量**：与总条数有关，不一定高 QPS |
| 重复消息 | 重传、重推、重连 SYNC 导致同一条多次到达 → 重复展示 | SDK 入站去重（seenIds）、出站 clientMsgId 幂等 | **可靠性/多路**：重传、重连就会发生，高并发时更易暴露 |
| 顺序与一致性 | 多路到达、弱网乱序，到达顺序 ≠ 发送顺序 | 服务端 seqId、前端按 seqId 排序、协议帧 seq | **多路/弱网**：与并发有一定关系，本质是顺序保证 |
| 服务端/网络压力 | 单用户刷屏打满服务端和广播 | 后端按用户限流、SDK 出站批处理 | **高并发**：单用户高 QPS 直接导致 |
| 大帧与传输 | 单帧过大 → 阻塞、超时、解析慢 | 大帧分片、可选 Protobuf | **大 payload**：单次体积大，高 QPS 时 Protobuf 更划算 |

## 一、SDK 层

**遇到的问题**

- 高频收消息时，每条都触发一次 `setState` 更新列表，渲染次数等于消息条数，主线程被打满，输入和滚动卡顿。
- 出站每条消息单独发一帧，WebSocket 帧数爆炸，服务端和网络压力大。
- 网络重传、服务端重推、重连 SYNC 会导致同一条消息多次到达，若不做去重会重复展示、重复写入。

**为什么这样设计**

- **入站/出站都走 MessageQueue**：把「每条消息一次更新」变成「按时间窗口或批次合并后再更新/再发送」，用**批处理**把更新频率和帧数降下来。
- **入站去重（seenIds + 时间窗口）**：
  - **数据结构**：`seenIds` 为 `Map<message.id, number>`，key 是消息 id，value 是该 id **首次**被入队时的时间戳（`Date.now()`）。
  - **时间窗口**：配置项 `deduplicationWindow`（默认 5s）。在「当前时间 − 窗口」之前写入的 seenIds 条目会被定期清理；清理后该 id 不再在 Map 里，视为**窗口外**。
  - **判定重复**：每次 `enqueueIncoming(message)` 时，若 `seenIds.has(message.id)` 为 true，则视为**重复**，**不**把该消息 push 进 `incoming`，直接返回 false；若为 false，则视为首次出现，执行 `seenIds.set(message.id, Date.now())` 并将消息 push 进 `incoming`，然后调用 `cleanupDedup()`。
  - **窗口外清理**：`cleanupDedup()` 在每次入队后执行，删除所有「value &lt; Date.now() − deduplicationWindow」的条目。这样同一 id 在超过窗口时长未再出现后，其记录被删掉，**该 id 再次到达时会被当作未见过而允许入队**（例如重连 SYNC 或延迟重推）；若在窗口内再次到达则只保留第一次，后续丢弃。
  - **去重只在入队处做**：只有通过上述检查的消息才会进入 `incoming`；后续的 flush、合并、按 seqId 排序、`onFlushIncoming` 派发到 store/UI 都只消费 `incoming`，因此**同一 id 在窗口内最多只会在列表里出现一次**，后续逻辑无需再判重，自然只看到一份。同时定期清理过期 id，避免 seenIds 无限增长。
- **出站用 clientMsgId 幂等**：服务端按 clientMsgId 去重，重试或重连重发不会产生重复消息。

**好处**

- 例如每秒 20 条消息：不批处理约 20 次列表更新；批处理后每 50ms 最多 1 次 flush，渲染次数和 CPU 明显下降。
- 出站多条合并为一帧 SEND_MESSAGE，降低 WebSocket 帧数，减轻服务端压力。
- 去重后同一条消息只展示一次，数据一致、体验稳定。

**如何解决高并发（小结）**

- **入站**：50ms 定时 flush + 每批最多 300 条，先入队再一次性交给 `onFlushIncoming` 更新 store 并派发；入队时按 message.id 经 seenIds（5s 窗口）去重。
- **出站**：同一队列按批调用 `onFlushOutgoing`，多条合并为一帧；pendingAck + 断线回滚保证不丢、不重；maxSize 背压防止队列无限增长。

## 二、前端层

**遇到的问题**

- 几千、上万条消息全量渲染，DOM 数量与消息条数线性增长，首屏慢、滚动卡、内存高。
- 列表 key 不稳定（如用 serverMsgId，ACK 后 id 变化）会导致列表重排、闪动、错位。
- 新消息到达时若总是自动滚到底，用户在看历史时会被打断；不滚又可能看不到新消息。

**为什么这样设计**

- **用 Virtuoso 做虚拟列表**：只渲染视口 + overscan（如 5 条），DOM 数量与总条数解耦，1 万条也只维持视口内约 20～30 个节点，从根上解决长列表性能问题。
- **列表 key 用 message.id（clientMsgId），不随 ACK 改**：ACK 后只更新 status/seqId/metadata，不替换 id，避免 key 变化导致整列表重排和错位。
- **followOutput 按是否在底部决定是否自动滚动**：在底部时新消息平滑滚到底；不在底部（用户在看历史）不自动滚动，减少无效滚动和重排。

**好处**

- 万级消息下滚动仍流畅，FPS 稳定，内存可控。
- 列表更新时 DOM 正确复用，不闪、不错位。
- 新消息与「看历史」的体验平衡：在底部跟到底，不在底部不打扰。

**如何解决高并发（小结）**

- **虚拟化**：MessageList / HistoryMessageList 使用 Virtuoso，`data={messages}`，只渲染视口 + overscan。
- **稳定 key**：`computeItemKey={(_, msg) => msg.id}`（clientMsgId）。
- **followOutput**：`followOutput={(isAtBottom) => (isAtBottom ? 'smooth' : false)}`。
- **历史按需加载**：滚动到顶部时 `loadMoreHistory()`，避免首屏一次性拉全量。

## 三、后端层

**遇到的问题**

- 单用户或少数用户疯狂发消息时，服务端处理、广播、存储都会被打满，影响其他用户和系统稳定性。
- 消息来源多路（实时推送、SYNC、历史），需要全局一致的顺序与去重依据。
- 单帧过大（大批量消息、大附件元数据）会导致传输阻塞、超时或解析慢。

**为什么这样设计**

- **按用户限流（滑动窗口）**：每用户每秒最多 N 条（如 20 条/秒），超限直接返回 rate_limit，从入口控制单用户 QPS，保护服务端和广播。
- **会话内 seqId 单调递增**：所有消息落库时分配 seqId，前端多路合并时按 seqId 排序，保证顺序一致；协议帧带 seq，便于顺序与幂等。
- **大帧分片**：单帧超过阈值（如 64KB）拆成 frag_meta + 多 chunk，接收端重组，避免单帧过大带来的阻塞和超时。

**好处**

- 限流后单用户无法拖垮服务端，其他用户体验不受影响。
- seqId + 帧 seq 让「顺序、去重、幂等重试」有统一依据，前后端一致。
- 大帧分片后传输和解析更可控，高 QPS 时还可选 Protobuf 减小体积、降低解析成本。

**如何解决高并发（小结）**

- **限流**：rateLimitMap 按 userId 存时间戳数组，滑动窗口内超过 N 条即拒绝并返回 rate_limit。
- **顺序与去重**：消息落库分配 seqId；处理 SEND_MESSAGE 时按 clientMsgId 幂等；帧带 seq。
- **传输**：大帧分片（frag_meta + chunks）；支持 JSON/Protobuf 双格式，高 QPS 可切 Protobuf。

## 四、一句话总结（给面试官）

**高并发场景下**：**SDK 层**用 MessageQueue 做入站/出站批处理 + seenIds 去重 + 背压，把「每条一次更新/一帧」压成「按批更新/按批发帧」，并保证不重、不乱序；**前端层**用 Virtuoso 虚拟列表 + 稳定 key（clientMsgId）+ followOutput，让万级消息下 DOM 恒定、滚动流畅；**后端层**用按用户滑动窗口限流 + seqId/帧 seq 保证顺序与幂等 + 大帧分片（和可选 Protobuf），保护服务端和网络。这样从前到后一起解决「更新频率、DOM 规模、重复、顺序、服务端压力、传输」几类问题，在万级消息、高频收发下仍能保持列表流畅、不重复、不乱序。

## 五、限流：滑动窗口 vs 固定窗口（面试可答）

**问：滑动窗口和固定窗口有什么区别？**

| 对比 | 固定窗口 | 滑动窗口 |
|------|----------|----------|
| **统计方式** | 按「自然秒/分钟」划分，每个窗口内独立计数 | 按「当前时刻往前 N 秒」内的请求数统计 |
| **边界突刺** | **有**：窗口边界处可「前一秒末 + 后一秒初」各打满限额，实际 2 倍 QPS 通过 | **无**：任意连续 1 秒内的请求数都被限制，不会在边界叠加 |
| **平滑度** | 边界处流量易突刺，限流不够平滑 | 更平滑，任意连续窗口内都受约束 |
| **精度** | 粗（按整秒/整分钟切） | 更精准（按真实时间间隔判断） |

**小结**：固定窗口有**边界突刺问题**（两窗口交界处可打出 2 倍限额），滑动窗口按「当前时刻往前 N 秒」统计，**更平滑、限流更精准**，本项目按用户限流采用滑动窗口（`checkRateLimit` 里 filter 掉 1 秒外的时间戳再判条数）。
