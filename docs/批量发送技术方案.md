# 批量发送技术方案

本文档描述 IM 系统**出站消息批量发送**的技术方案，即：将多条消息合并为一个 WebSocket 帧发送，减少帧数、降低网络与序列化开销。

---

## 一、背景与目标

### 1.1 背景

在高 QPS 场景下（如压测、群聊刷屏），MessageQueue 每 50ms 批量 flush 出站消息。原实现中，**每条消息单独发一个 WebSocket 帧**：

```
flush → handleOutgoingBatch([msg1, msg2, msg3])
      → ws.send({ type: 'send_message', payload: msg1 })  // 帧 1
      → ws.send({ type: 'send_message', payload: msg2 })  // 帧 2
      → ws.send({ type: 'send_message', payload: msg3 })  // 帧 3
```

**问题**：
- N 条消息 = N 次 `ws.send()`、N 次 `JSON.stringify()`
- 帧数多，TCP 分片与网络开销增加

### 1.2 目标

- 多条消息合并为**一个 WebSocket 帧**发送，payload 为消息数组
- 单条消息仍保持原有格式（单对象），兼容旧逻辑
- 服务端逐条处理、逐条限流、逐条回 ACK，语义不变

## 二、协议设计

### 2.1 SEND_MESSAGE 帧格式

| 字段 | 类型 | 说明 |
|------|------|------|
| type | string | 固定 `"send_message"` |
| seq | number | 帧序列号 |
| timestamp | number | 时间戳 |
| payload | Message \| Message[] | **单条为对象，批量为数组** |

**单条消息示例**：
```json
{
  "type": "send_message",
  "seq": 48,
  "timestamp": 1739256789123,
  "payload": {
    "id": "msg-1771178784379-xcolt8",
    "conversationId": "conv-bot-xxx",
    "content": "hello",
    "type": "text",
    "status": "sending",
    "senderType": "user",
    "senderId": "user1",
    "senderName": "You",
    "timestamp": 1739256789000
  }
}
```

**批量消息示例**：
```json
{
  "type": "send_message",
  "seq": 49,
  "timestamp": 1739256789200,
  "payload": [
    { "id": "msg-1", "content": "msg1", ... },
    { "id": "msg-2", "content": "msg2", ... },
    { "id": "msg-3", "content": "msg3", ... }
  ]
}
```

### 2.2 message_ack 批量格式

服务端在批量处理完一批用户消息后，**一次发送** `message_ack`，payload 支持：

| 情况 | payload 类型 |
|------|--------------|
| 成功 1 条 | `{ clientMsgId, serverMsgId, seqId }` |
| 成功 N 条 | `[{ clientMsgId, serverMsgId, seqId }, ...]` |

客户端统一 `items = Array.isArray(payload) ? payload : [payload]` 后逐条处理。

**一次 ack 多少条？** 与「收到的 send_message 批次」一一对应：每条 `message_ack` 帧中的 ack 数量 = 该批 `send_message` 中**成功通过限流并入库**的消息数。MessageQueue 的 `batchSize` 为 30，故单次 flush 最多 30 条，即一条 `message_ack` 中最多 30 个 ack；若部分被限流，则数量更少。

| 客户端发送 | 限流 | message_ack payload 条数 |
|------------|------|--------------------------|
| 1 条 | 0 | 1（单对象） |
| 10 条 | 0 | 10 |
| 30 条 | 0 | 30 |
| 30 条 | 10 条被限流 | 20 |

### 2.3 兼容性

- 旧客户端：继续发 `payload: Message`，服务端 `Array.isArray` 为 false，视为单元素数组处理
- 新客户端：单条发对象，多条发数组
- 服务端：统一 `messages = Array.isArray(payload) ? payload : [payload]` 后逐条处理
- message_ack：单条与批量 payload 格式兼容，客户端均可解析

## 三、客户端实现

### 3.1 修改位置

`src/sdk/IMClient.ts` → `handleOutgoingBatch`

### 3.2 实现逻辑

```typescript
private async handleOutgoingBatch(messages: Message[]): Promise<void> {
  if (messages.length === 0) return;
  if (messages.length === 1) {
    this.wsManager.send(FrameType.SEND_MESSAGE, messages[0]);
  } else {
    this.wsManager.send(FrameType.SEND_MESSAGE, messages);
  }
}
```

| 分支 | payload 类型 | 帧数 |
|------|--------------|------|
| 0 条 | 不发送 | 0 |
| 1 条 | Message（对象） | 1 |
| N 条 | Message[]（数组） | 1 |

### 3.3 调用链

```
用户 sendMessage / 压测快速发送
  → messageQueue.enqueueOutgoing(msg) × N
  → [50ms 定时 flush]
  → flushOutgoing() 取 batch = splice(0, batchSize)
  → onFlushOutgoing(batch) 即 handleOutgoingBatch(messages)
  → wsManager.send(SEND_MESSAGE, messages.length === 1 ? messages[0] : messages)
  → ws.send(JSON.stringify(frame))
```

## 四、服务端实现

### 4.1 修改位置

`server/ws-handler.ts` → `case FrameType.SEND_MESSAGE`

### 4.2 实现逻辑

1. **统一为数组**：`const messages = Array.isArray(frame.payload) ? frame.payload : [frame.payload]`
2. **逐条处理**：`for (const msg of messages) { ... }`
3. **逐条限流**：每条消息独立调用 `checkRateLimit(userId)`，超限则 `send(ERROR)` 并 `continue`
4. **逐条 ACK**：每条消息处理完成后发送 `MESSAGE_ACK`，回复格式不变
5. **会话状态**：每轮循环内 `conv = db.getConversation(convId)` 获取最新会话，支持上一轮 `handleRequestAgent` 等变更

### 4.3 关键代码结构

```typescript
case FrameType.SEND_MESSAGE: {
  const messages = Array.isArray(frame.payload)
    ? (frame.payload as Message[])
    : [frame.payload as Message];
  for (const msg of messages) {
    if (!checkRateLimit(userId)) {
      send(ws, FrameType.ERROR, { code: "rate_limit", ... });
      continue;
    }
    const conv2 = db.getConversation(convId);
    if (!conv2) continue;
    // ... 原有单条处理逻辑（toInsert、bot/agent 回复、ACK 等）
  }
  break;
}
```

### 4.4 限流与 ACK 语义

| 场景 | 行为 |
|------|------|
| 批量 10 条，前 5 条通过、第 6 条限流 | 前 5 条入库并回 ACK，第 6 条回 ERROR，后 4 条继续处理（各自独立限流） |
| 第 3 条触发转人工 | 处理完第 3 条后 conv 变更，第 4 条及以后按新 phase 处理 |
| 某条消息格式异常 | 可 try/catch 单条，不影响其余消息（当前实现未做，可按需扩展） |

## 五、数据流

### 5.1 批量发送端到端

```
[客户端]
  sendMessage × 30 (50ms 内)
    → enqueueOutgoing × 30
    → [50ms flush]
    → handleOutgoingBatch([m1..m30])
    → ws.send({ type: 'send_message', payload: [m1..m30] })   // 1 帧

[网络]
  1 个 WebSocket 帧，payload 为 JSON 数组

[服务端]
  handleFrame(SEND_MESSAGE)
    → messages = [m1..m30]
    → for msg in messages:
        checkRateLimit → 通过则入库、收集 ack，否则回 ERROR
    → send(MESSAGE_ACK, acksToSend)   // 1 帧，payload 为 [ack1, ack2, ...]
    → send(ERROR) × 限流条数（若有）
    → send(MESSAGE, botReply) × 需要回复的条数

[客户端]
  收到 1 个 MESSAGE_ACK 帧（payload 为数组）
    → 遍历 payload，messageQueue.onAck(clientMsgId) × N
    → 更新消息 status 为 SENT
```

### 5.2 帧数对比（批量 10 条同 flush）

| 方向 | 旧实现 | 新实现 |
|------|--------|--------|
| send_message | 10 帧 | 1 帧 |
| message_ack | 10 帧 | 1 帧 |
| **合计** | 20 帧 | 2 帧 |

## 六、与现有模块的关系

| 模块 | 关系 |
|------|------|
| MessageQueue | flush 仍按 batchSize、flushInterval 触发，handleOutgoingBatch 入参不变，仅发送方式改为单帧数组 |
| WebSocketManager | 无改动，`send(type, payload)` 已支持任意 payload |
| ws-handler 限流 | 仍按条限流，批量时逐条 checkRateLimit |
| message_ack 处理 | 客户端仍逐条接收 ACK，逻辑不变 |
| 高 QPS 压测页 | 可直接验证：发 30 条 30ms 间隔，网络面板应看到单帧 `payload: [...]` |

## 七、验证方式

### 7.1 网络面板验证

1. 打开 `/stress`，连接后设置「发送 30 条」「间隔 30ms」
2. 打开 DevTools → Network → 选中 WebSocket 连接 → Messages
3. 点击「开始高 QPS 压测」
4. **预期**：出现 `↑ send_message` 帧，展开后 `payload` 为数组 `[{...}, {...}, ...]`，帧数为 1（若 30 条在同一 flush 窗口）

### 7.2 功能验证

- 单条发送：聊天窗口正常发一条，行为与改造前一致
- 批量发送：压测 30 条，消息列表应正常展示，部分可能因限流显示失败
- 限流统计：压测页「限流」数量与网络面板中 `ERROR` 帧数量一致

| 项 | 说明 |
|------|------|
| 协议 | SEND_MESSAGE payload 支持 `Message \| Message[]`，向后兼容 |
| 客户端 | handleOutgoingBatch 多条时发送 `payload: messages` |
