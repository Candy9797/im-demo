# 消息已读/未读处理说明

本文档说明本 IM 项目中消息已读、未读状态的完整处理流程。

---

## 一、数据模型

### 1.1 消息状态 (MessageStatus)

```typescript
// src/sdk/types.ts
MessageStatus: SENDING | SENT | DELIVERED | READ | FAILED
```

- **SENDING**：客户端已发送，等待 ACK
- **SENT**：服务端已确认
- **DELIVERED**：已送达（本项目与 SENT 同义）
- **READ**：收件人已读
- **FAILED**：发送失败

### 1.2 已读元数据 (metadata.readBy)

```typescript
metadata.readBy?: string[]  // 已读用户的 userId 列表
```

用于记录哪些用户已读过该消息，支持扩展（如按设备区分）。

---

## 二、未读判定规则

**未读**：来自 Bot/Agent（非用户自己）、且 `status !== "read"` 的消息。

```typescript
// src/store/chatStore.ts - countUnread()
messages.filter(m =>
  m.senderType !== SenderType.USER &&   // 排除用户自己发的
  m.senderType !== SenderType.SYSTEM && // 排除系统消息
  m.status !== "read"                   // 未标记为已读
).length
```

- 用户自己发的消息不参与未读计数
- 系统消息（如 Welcome）不参与
- 仅当 `status === "read"` 视为已读

---

## 三、已读触发流程

### 3.1 触发条件

当 **Bot/Agent 消息** 进入可视区域时，将 messageId 加入待上报队列，批量上报已读。

```
MessageList 传入 onMessageVisible
    ↓
MessageItem 用 IntersectionObserver 监听
    ↓
消息进入视口 ≥ 50% → 调用 onVisible(messageId)
    ↓
pendingReadRef.add(messageId)
    ↓
达到 20 条 或 debounce 200ms 后 → markAsRead(ids)
```

### 3.2 代码位置

| 环节 | 文件 | 说明 |
|------|------|------|
| 可见性检测 | `MessageItem.tsx` | `IntersectionObserver`，`threshold: 0.5` |
| 过滤发送者 | `MessageItem.tsx` | `message.senderType === SenderType.USER` 时不同调 onVisible |
| 批量聚合 | `MessageList.tsx` | `pendingReadRef` 收集，debounce 200ms / 满 20 条即 flush |
| Store 调用 | `chatStore.ts` | `markAsRead(ids) → client.markAsRead(ids)` |
| 网络发送 | `IMClient.ts` | `wsManager.send(FrameType.MARK_READ, { messageIds })` |

---

## 四、服务端处理

### 4.1 收到 MARK_READ 帧

```typescript
// server/ws-handler.ts
case FrameType.MARK_READ:
  for (const msgId of messageIds) {
    db.updateMessageStatus(msgId, "read");           // 1. 更新 status
    const row = db.getMessage(msgId);
    if (row) {
      meta.readBy.push(userId);                      // 2. 更新 readBy
      db.updateMessageMetadata(msgId, JSON.stringify(meta));
    }
  }
  sendToUser(userId, FrameType.READ_RECEIPT, { messageIds, readBy: userId });  // 3. 回执
```

### 4.2 DB 更新

- `updateMessageStatus(msgId, "read")`：更新 `messages.status`
- `updateMessageMetadata`：更新 `messages.metadata` 中的 `readBy` 数组

### 4.3 READ_RECEIPT 回执

服务端向**当前用户**下发 READ_RECEIPT，用于：
1. 更新本端 UI（status → read）
2. 多端时同步到同用户其他设备（当前为单端模型）

---

## 五、客户端收到 READ_RECEIPT 后

```typescript
// chatStore.ts - client.on(SDKEvent.READ_RECEIPT)
set((s) => ({
  messages: s.messages.map((m) => {
    if (!messageIds.includes(m.id)) return m;
    const readByList = meta.readBy ?? [];
    if (readByList.includes(readBy)) return m;  // 已记录，跳过
    return {
      ...m,
      status: "read",
      metadata: { ...meta, readBy: [...readByList, readBy] },
    };
  }),
}));
```

- 找到对应消息，更新 `status` 为 `"read"`
- 在 `metadata.readBy` 中追加当前用户
- 触发 Store 更新 → 未读计数减少 → 角标更新

---

## 六、完整数据流

```
用户滚动列表，Bot 消息进入视口
        ↓
IntersectionObserver 触发
        ↓
markAsRead([msgId])
        ↓
IMClient.markAsRead → WebSocket 发送 MARK_READ
        ↓
服务端: updateMessageStatus + updateMessageMetadata
        ↓
服务端: sendToUser(READ_RECEIPT)
        ↓
客户端: READ_RECEIPT → chatStore set({ messages: ... })
        ↓
countUnread() 减少 → 角标更新
MessageItem StatusIcon 显示 ✓✓ (Read)
```

---

## 七、未读角标展示

| 位置 | 条件 | 实现 |
|------|------|------|
| ChatTrigger | `!isOpen && unreadCount > 0` | 入口按钮红点 |
| Minimized 栏 | `isMinimized && unreadCount > 0` | 最小化栏旁数字 |

`unreadCount = countUnread(messages, auth?.userId)`

---

## 八、注意事项

1. **仅对收到的消息上报**：用户自己发的消息不会触发 markAsRead
2. **client_msg_id 兼容**：服务端 `getMessage(msgId)` 支持按 `client_msg_id` 查找（需传入 convId），当前流程中 markAsRead 的消息均为服务端下发的，已带 server id
3. **无乐观更新**：markAsRead 后等待 READ_RECEIPT 才更新 UI，存在约 1 RTT 延迟
4. **单端模型**：当前 `kickOthers=true` 时新连接会踢掉旧连接，无多端在线；若支持多端，READ_RECEIPT 需广播给同用户所有连接
