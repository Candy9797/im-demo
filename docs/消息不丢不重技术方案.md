# IM 系统消息不丢不重技术方案

本文档描述项目中保证消息**不丢失**、**不重复**的机制与实现。

---

## 一、问题与目标

### 1.1 典型场景

| 场景 | 可能导致丢消息 | 可能导致重消息 |
|------|----------------|----------------|
| 网络抖动、断线重连 | 发送时连接断开，消息未到达服务端 | 客户端重试，服务端重复处理 |
| 服务端重推 | - | 重连后拉取断线期间消息，与本地已有重叠 |
| 多端同步 | 某端离线期间消息未同步 | 多端同时收到同一消息 |
| 乐观更新 + ACK | 本地先展示，ACK 迟到或丢失 | 同一消息在 ACK 和 frame_in 中重复 |
| 历史分页 / SYNC | 拉取时与本地边界重叠 | afterSeqId 不准导致重复区间 |

### 1.2 设计目标

- **不丢**：发送的消息最终能到达服务端并持久化；收消息时断线期间的消息能在重连后补齐
- **不重**：同一逻辑消息在 UI 和存储中只出现一次
- **有序**：消息按时间/seq 正确排序，不乱序

## 二、整体策略

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           防丢策略                                            │
│  • 客户端乐观更新 + ACK 确认                                                  │
│  • 出站队列 + 重试                                                           │
│  • 服务端持久化 (SQLite)                                                      │
│  • 客户端 IndexedDB 离线兜底                                                   │
│  • 重连后 SYNC / auth_ok 补拉                                                 │
└─────────────────────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────────────────────┐
│                           防重策略                                            │
│  • client_msg_id / server_msg_id 映射，ACK 幂等                               │
│  • 入站 seenIds 窗口去重                                                      │
│  • 服务端按 client_msg_id 查重（可选）                                         │
│  • Store / sync 合并时按 message.id 去重                                      │
│  • seq_id 单调递增，按 seq 排序                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 三、防丢机制

### 3.1 发送侧：乐观更新 + ACK + 重试

**流程**：
1. 用户发送 → `sendMessage` 立即构造 Message（`id` 为 client_msg_id，`status: SENDING`）推入列表
2. MessageQueue 出队 → `handleOutgoingBatch` → `ws.send(SEND_MESSAGE, msg)`
3. 服务端入库 → 返回 `message_ack { clientMsgId, serverMsgId, seqId }`
4. 客户端收到 ACK → 按 `clientMsgId` 找到本地消息，更新 `id = serverMsgId`、`status = SENT`，写入 IndexedDB

**防丢要点**：
- 乐观更新：用户无感知延迟，消息先展示
- ACK 匹配：通过 `client_msg_id` 唯一关联，即使消息乱序也能正确更新
- 出站队列：MessageQueue 断线时 `pause`，重连后 `resume`，未 ACK 的消息继续发送（当前实现为内存队列，刷新会丢；可扩展为 IndexedDB 持久化）

### 3.2 出站重试

**MessageQueue 配置**：
```typescript
retryAttempts: 3,
retryDelay: 1000,  // 指数退避基数
```

**行为**：
- `handleOutgoingBatch` 抛出异常时，将本批消息重新入队
- 达到 `retryAttempts` 后标记 `status: FAILED`，推入入队触发 UI 展示失败态

### 3.3 服务端持久化

- **SQLite**：每条消息入库，`seq_id` 单调递增，支持 `getMessagesAfter(convId, afterSeqId)` 按 seq 区间拉取
- **client_msg_id**：入库时保存，用于 ACK 幂等及反应/已读等按消息 id 查找（支持 client_msg_id 或 server id）

### 3.4 客户端离线存储

- **IndexedDB**：`saveMessage` / `saveMessages` 写入，按 `conversationId` 分区
- **冷启动**：`auth_ok` 时若服务端无消息，从 IndexedDB 拉取本地缓存展示
- **ACK 后写入**：收到 ACK 更新 id/status 后立即 `saveMessage`，保证离线恢复时状态正确

### 3.5 重连后补拉

- **auth_ok**：连接成功时服务端下发 `conversationId`、`messages`（最近 100 条），客户端覆盖/合并
- **SYNC**：客户端可发 `SYNC { afterSeqId }`，服务端返回 `seq_id > afterSeqId` 的消息
- **LOAD_HISTORY**：上拉加载更早历史，`beforeSeqId` 分页

**afterSeqId 的确定**：本地 `getMaxSeqId(conversationId)` 或 `Math.max(...messages.map(m => m.seqId))`，取最大值作为断点。若本地有未 ACK 的乐观消息，其 seqId 尚未确定，需在合并时注意边界处理。

## 四、防重机制

### 4.1 client_msg_id / server_msg_id 幂等

**设计**：
- 客户端发送时自带 `message.id`（如 `msg-${Date.now()}-${random}`），即 client_msg_id
- 服务端入库时生成 `server_msg_id`，表字段 `client_msg_id` 保存客户端 id
- ACK 返回 `{ clientMsgId, serverMsgId }`，客户端用 clientMsgId 匹配本地消息，更新为 serverMsgId

**幂等**：
- 同一 client_msg_id 多次发送（如重试），服务端可按 client_msg_id 查重，只插入一次（当前实现未显式查重，依赖「每条消息只发一次」；若重试会重复入库，需扩展）
- ACK 时客户端按 clientMsgId 查找，只更新一条，不会重复插入

### 4.2 入站去重：seenIds 窗口

**MessageQueue**：
```typescript
private seenIds: Map<string, number> = new Map();  // id -> timestamp
deduplicationWindow: 5000,  // 5 秒
```

**流程**：
1. 收到 `frame_in` → `enqueueIncoming(msg)`
2. `if (isDuplicate(msg.id)) return false` → 已见过则丢弃
3. `seenIds.set(msg.id, Date.now())`，定期 `cleanupDedup()` 清理超过 5 秒的条目

**覆盖场景**：服务端重推、网络重复包、同一消息在多帧中出现

### 4.3 合并时按 id 去重

**chatStore**：
```typescript
// MESSAGE_SENT
if (s.messages.some((m) => m.id === message.id)) return s;

// MESSAGE_RECEIVED / MESSAGE_BATCH_RECEIVED
const ids = new Set(s.messages.map((m) => m.id));
const newOnes = batch.filter((m) => !ids.has(m.id));

// HISTORY_LOADED
const ids = new Set(s.messages.map((m) => m.id));
const merged = [...newMsgs.filter((m) => !ids.has(m.id)), ...s.messages].sort(...);
```

**IMClient**：
```typescript
// frame_in
if (!this.conversation.messages.some((m) => m.id === msg.id)) {
  this.conversation.messages.push(msg);
  this.messageQueue.enqueueIncoming(msg);
}

// sync_response
const newMsgs = messages.filter((msg) => !this.conversation.messages.some((m) => m.id === msg.id));

// history_response
const newMsgs = messages.filter((m) => !this.conversation.messages.some((x) => x.id === m.id));
```

**原则**：所有「合并新消息」的路径都按 `message.id` 去重，避免重复展示。

### 4.4 顺序保证：seq_id

- **服务端**：`nextSeqId(convId)` 基于 `MAX(seq_id)+1`，保证会话内单调递增
- **拉取**：`getMessagesAfter(convId, afterSeqId)` 按 `seq_id > afterSeqId ORDER BY seq_id ASC`
- **展示**：客户端按 `(seqId ?? timestamp)` 排序

## 五、数据流示意

### 5.1 发送流程

```
用户输入 → sendMessage(content)
    │
    ├─ createMessage() → msg (id=client_msg_id, status=SENDING)
    ├─ conversation.messages.push(msg)
    ├─ messageQueue.enqueueOutgoing(msg)
    └─ emit(MESSAGE_SENT)
    │
    ▼
MessageQueue flushOutgoing
    │
    ├─ handleOutgoingBatch(msgs)
    └─ ws.send(SEND_MESSAGE, msg)  [每条约一条]
    │
    ▼
服务端 handleFrame(SEND_MESSAGE)
    │
    ├─ nextSeqId(convId)
    ├─ insertMessages([{ msgId: server_msg_id, clientMsgId: msg.id, ... }])
    └─ send(message_ack, { clientMsgId, serverMsgId, seqId })
    │
    ▼
客户端 message_ack
    │
    ├─ find(msg => msg.id === clientMsgId)
    ├─ msg.id = serverMsgId; msg.status = SENT
    ├─ emit(MESSAGE_STATUS_UPDATE)
    └─ saveMessage(msg)  // IndexedDB
```

### 5.2 接收流程

```
服务端 send(MESSAGE, msg)
    │
    ▼
WebSocketManager handleFrame → emit("frame_in", frame)
    │
    ▼
IMClient frame_in
    │
    ├─ if (!messages.some(m => m.id === msg.id))  // 去重
    │     conversation.messages.push(msg)
    │     messageQueue.enqueueIncoming(msg)
    │
    ▼
MessageQueue enqueueIncoming
    │
    ├─ if (isDuplicate(msg.id)) return false   // seenIds 去重
    ├─ seenIds.set(msg.id, Date.now())
    └─ incoming.push(msg)
    │
    ▼
flushIncoming (每 flushInterval ms)
    │
    └─ handleIncomingBatch(batch)
           ├─ emit(MESSAGE_RECEIVED / MESSAGE_BATCH_RECEIVED)
           └─ saveMessages(batch)  // IndexedDB
```

## 六、关键参数

| 模块 | 参数 | 默认值 | 说明 |
|------|------|--------|------|
| MessageQueue | deduplicationWindow | 5000 | 去重窗口（ms），超时从 seenIds 清理 |
| MessageQueue | retryAttempts | 3 | 出站失败重试次数 |
| MessageQueue | flushInterval | 100 | 入站批处理间隔（ms） |
| IndexedDBStore | WRITE_DEBOUNCE_MS | 80 | saveMessage 防抖间隔 |
| IndexedDBStore | MAX_BUFFER_SIZE | 50 | 缓冲条数，达量即写 |
| ws-handler | INITIAL_LOAD_LIMIT | 100 | auth_ok 下发消息条数 |
| ws-handler | HISTORY_PAGE_SIZE | 50 | 历史分页大小 |

## 七、已知限制与改进方向

### 7.1 当前限制

| 项目 | 说明 |
|------|------|
| 出队队列不持久 | 断线期间待发消息在内存中，刷新丢失；可扩展 IndexedDB 持久化出队 |
| 服务端未按 client_msg_id 入库去重 | 若客户端重试发送同一 client_msg_id，可能重复入库；可加唯一约束或 upsert |
| afterSeqId 与乐观消息 | 本地有未 ACK 消息时，其 seqId 未定，SYNC 的 afterSeqId 可能偏小，拉取到重叠区间，依赖 seenIds 去重 |
| seenIds 窗口 5s | 超窗口的重复消息不会去重；可调大或改为按会话持久化已见 id |

### 7.2 改进方向

1. **服务端幂等**：对 `(conversation_id, client_msg_id)` 建唯一索引或先 `SELECT` 再 `INSERT`，避免重试导致重复
2. **出队持久化**：断线时将 outgoing 队列写入 IndexedDB，重连后恢复发送
3. **ACK 超时重发**：对长期未 ACK 的消息标记失败或重发（当前仅依赖 flush 失败重试）
4. **read_after_write**：关键消息发送后主动发 SYNC 拉取，校验是否落库
5. **多端一致性**：多设备登录时，通过 lastSeqId 同步，避免漏拉或重复

---

## 八、小结

| 维度 | 机制 |
|------|------|
| **防丢** | 乐观更新、ACK 确认、出队重试、服务端持久化、IndexedDB 兜底、重连补拉 |
| **防重** | client_msg_id 映射、seenIds 窗口去重、合并按 id 去重、seq_id 排序 |
| **有序** | seq_id 单调递增、getMessagesAfter 按 seq 拉取、展示按 seq/timestamp 排序 |

以上机制组合，在常规网络与负载下可实现消息不丢不重的目标；在极端场景（长时间断网、多端并发、服务端故障）下，需结合业务做进一步加固与监控。
