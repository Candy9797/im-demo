# IM 高级功能：技术原理与面试要点

面向前端/IM 架构师面试，按「一句话概括 + 技术原理 + 实现要点 + 面试官可能问」组织，可直接说出口、工程上真实存在。

---

## 1. 消息可靠性保障（必问）

### 1.1 一句话

**消息必达**：通过 ACK + 超时重试 + 去重，保证「发出去的消息要么对方收到，要么客户端明确失败可重发」。

### 1.2 技术原理

- **ACK 机制**：客户端发消息带 `clientMsgId`，服务端落库后回 ACK（带 `serverMsgId`、`seq`）。客户端收到 ACK 才把该条从「发送中」改为「已发送」，未在超时内收到则重发。
- **失败重试**：超时（如 10s）未 ACK 则重发同一条（幂等键用 `clientMsgId`），限制重试次数（如 3 次），超过则标记失败、允许用户「重发」。
- **消息去重**：服务端用 `clientMsgId` 幂等，同一 `clientMsgId` 只落库一次；多端/重试导致重复请求时，返回同一条 `serverMsgId`，客户端按 `clientMsgId` 去重展示。
- **有序**：服务端为会话内消息分配**单调递增 seq**，下发与拉取都带 seq；客户端按 seq 排序展示，乱序到达时先缓存再重排。

### 1.3 实现要点

- 客户端：待发队列 + 已发未 ACK 映射（clientMsgId → 发送时间），定时扫描超时并重发；收到 ACK 后更新本地消息的 serverMsgId/seq/status。
- 服务端：写库前查 clientMsgId 是否已存在，存在则直接返回已有记录；写库后推 ACK 与 MESSAGE 下行（推送给接收方）。

### 1.4 面试官可能问

- **Q：为什么既要 clientMsgId 又要 serverMsgId？**  
  A：clientMsgId 做幂等与去重、重试不产生重复消息；serverMsgId 做服务端唯一标识、拉历史/引用/已读都按 serverMsgId 关联。
- **Q：重试时怎么避免同一条消息在 UI 上出现两次？**  
  A：发时就用 clientMsgId 占位展示；收到 ACK 或重试成功只更新同一条的 status/serverMsgId，不追加新条。
- **Q：弱网下大量重试会不会把服务打挂？**  
  A：客户端限重试次数、指数退避；服务端限流 + clientMsgId 幂等，重复请求不重复写库。

## 2. 消息同步机制

### 2.1 一句话

**多端同步**：通过「服务端为权威 + 增量拉取 + 本地 merge」保证各端会话列表、未读、草稿、消息顺序一致。

### 2.2 技术原理

- **增量同步**：拉取时带「上次同步到的最大的 seq / 时间戳」，服务端只返回该会话在此之后的增量消息或摘要；避免每次全量拉全量。
- **按会话/按用户**：未读、已读游标、草稿按 conversationId 存；多端上报「已读到哪条」「草稿内容」，服务端合并后下发给其他端。
- **离线消息**：用户上线或拉会话时，按「lastSeq / 时间范围」拉取离线期间的消息；可做分页（按时间片或条数），避免一次拉爆。

### 2.3 实现要点

- 服务端：会话维度维护 maxSeq、每个用户在该会话的 lastReadSeq / 草稿 / 置顶/免打扰；增量接口：入参 lastSeq，返回 seq > lastSeq 的消息与状态变更。
- 客户端：本地存 lastSyncedSeq（或 per-conversation），拉取后 merge 进列表、更新 lastSyncedSeq；草稿/已读变更时上报服务端并写本地。

### 2.4 面试官可能问

- **Q：多端同时改草稿怎么处理？**  
  A：以「最后写入」或「版本号/时间戳」胜出，服务端合并后下发；或规定仅一端可编辑草稿（如仅 PC），避免冲突。
- **Q：增量拉取时服务端消息还没落库完，客户端漏消息怎么办？**  
  A：拉取接口返回「当前会话最大 seq」；若客户端本地 maxSeq 小于服务端返回的，可再拉一次或拉该区间补全。
- **Q：离线消息量特别大怎么办？**  
  A：按时间分片拉（如每次 7 天）、按条数分页；优先拉最近 N 条，更早的按需「加载更多」。

## 3. 已读回执 & 会话状态

### 3.1 一句话

**已读/送达**：服务端维护「每条消息的已读名单」和「会话维度的已读游标」，通过下行推送和拉取接口同步给多端；**正在输入**由客户端实时上报、服务端转发。

### 3.2 技术原理

- **单条已读**：读某条消息时上报 messageId（或 serverMsgId），服务端在该条上记录「谁已读」；需要展示已读名单时拉取或推送该信息。
- **会话已读**：维护「用户 A 在会话 C 已读到 seq=N」；未读数 = 该会话中 seq > N 且非自己发的消息数；多端同步 lastReadSeq 即可同步未读。
- **送达/已读双状态**：sent = 服务端已落库并 ACK；delivered = 对端设备已收到（可选，依赖对端上报）；read = 对端已读（同上，按条或按会话游标）。
- **正在输入**：输入时上报 typing_start，停顿时或发送后上报 typing_end；服务端转发给会话内其他人，前端展示「对方正在输入…」。可节流（如 3s 内只发一次）、防抖（停 2s 再发 end）。

### 3.3 实现要点

- 服务端：消息表或单独「已读表」记录 (userId, messageId/conversationId, seq)；会话表或用户-会话表存 lastReadSeq；typing 事件广播给会话内成员。
- 客户端：进入会话/滚动到某条时上报已读；收到 typing 事件更新 UI；未读数 = 服务端下发的数字或本地用 lastReadSeq 计算。

### 3.4 面试官可能问

- **Q：已读回执流量大怎么办？**  
  A：合并上报（进入会话一次报「已读到当前最大 seq」）、节流；服务端可做「仅同步最后一条已读」或按条数阈值同步。
- **Q：群聊已读要展示「谁已读」怎么做？**  
  A：按条存 readBy userId 列表；拉消息详情或拉「该条已读名单」接口；大群可只展示数量或部分头像，不全量拉。
- **Q：typing 会不会很刷屏？**  
  A：节流 + 短 TTL（如 5s 无更新自动置为 false）；仅在同一会话、同一设备内展示，不落库。

## 4. 消息撤回、编辑、引用

### 4.1 一句话

**撤回/编辑**：服务端做权限与时效校验，更新消息状态或内容并广播；**引用**：消息体带被引用消息的 id 与摘要，展示时拉原消息或用摘要渲染。

### 4.2 技术原理

- **撤回**：客户端发撤回请求（带 messageId）；服务端校验（是否本人/管理员、是否超时如 2 分钟）；通过则把该条标为「已撤回」或写撤回记录，内容可置空或替换为「已撤回」占位，并推送 MESSAGE_RECALL 给会话内所有人；多端拉取时用「已撤回」态覆盖原内容。
- **编辑**：客户端发编辑请求（messageId + 新 content）；服务端校验后更新内容与 editTime，推送 MESSAGE_UPDATE；多端 merge 时用新内容覆盖同 messageId。
- **引用/回复**：发送时在消息 metadata 里带 quote：{ messageId, senderName, content 摘要, type }；展示时根据 messageId 拉原消息（若在本地）或仅用摘要渲染；长对话可做「跳转到原消息」定位。

### 4.3 实现要点

- 服务端：撤回/编辑走单独协议或字段，写库更新 + 推送；引用只存元数据，不重复存被引用消息全文（可选存 content 摘要便于离线展示）。
- 客户端：收到 RECALL/UPDATE 更新本地该条状态或内容；引用块可点击 scrollToMessage(messageId)。

### 4.4 面试官可能问

- **Q：撤回后其他人已经截图/转发怎么办？**  
  A：产品上只能做到「会话内不再展示原内容」，无法消除已流出副本；可配合内容安全与举报。
- **Q：编辑历史要不要留？**  
  A：看产品：要则存版本表或 editHistory 数组；不要则只保留最新内容。
- **Q：被引用的消息被撤回了，引用块怎么展示？**  
  A：保留「引用」壳 + 展示「原消息已撤回」或仅显示发送者；不展示原内容。

## 5. 会话管理高级能力

### 5.1 一句话

**置顶/免打扰/漫游/搜索**：服务端按用户+会话存配置（置顶、免打扰、lastReadSeq 等）；消息漫游即按会话+时间拉历史；搜索走全文索引 + 过滤条件。

### 5.2 技术原理

- **置顶、免打扰**：用户-会话配置表，字段如 isPinned、muted；多端同步该配置，列表排序与角标按此渲染。
- **消息漫游**：换设备后拉取会话列表 + 各会话 lastSeq；再按需拉各会话历史（增量或分页），保证「换设备不丢记录」。
- **云端搜索**：消息落库后写入搜索引擎（ES/Meilisearch 等），按 userId/conversationId + 关键词 + 时间/类型筛选；结果返回 messageId 列表，客户端再按需拉详情或摘要。
- **未读数精算**：以服务端 lastReadSeq 为准；未读数 = 当前会话 maxSeq - lastReadSeq（仅统计非自己、未撤回的消息）；已读上报后服务端更新 lastReadSeq 并下发，避免多端各自算错。

### 5.3 实现要点

- 服务端：用户会话配置表；消息表按会话+seq 索引；搜索用 ES 等，建索引时包含 content、type、time、conversationId、userId。
- 客户端：拉会话列表时带配置与未读数；搜索请求带 keyword、conversationId、时间范围，结果高亮并支持跳转。

### 5.4 面试官可能问

- **Q：未读数多端不一致怎么查？**  
  A：以服务端为唯一真相；客户端仅展示服务端下发的未读数字；若不一致多为上报时机或拉取顺序问题，可加「未读校正」接口按服务端强制刷一次。
- **Q：消息漫游保留多久？**  
  A：看合规与成本：如 7 天/30 天/永久；过期可归档或只保留摘要，全文可做冷存储。
- **Q：搜索如何做高亮与性能？**  
  A：服务端返回命中的 messageId + 高亮片段（如 ES highlight）；前端只渲染当前页结果，大结果集分页。

## 6. 多媒体与富媒体消息

### 6.1 一句话

**图片/视频/文件**：上传走 OSS 拿 URL，消息只存 URL；支持缩略图、断点续传、秒传；**富媒体**：表情/贴纸为 type+id，卡片/链接为结构化数据 + 服务端预览或前端解析。

### 6.2 技术原理

- **上传**：客户端传文件（分片可选），服务端或直传 OSS 返回 url；消息体存 url + 可选 thumbnail、size、fileName；大文件做分片上传、断点续传、秒传（hash 命中则直接返回已有 url）。
- **缩略图**：服务端生成或前端生成小图 url，消息里带 thumbnail；列表与预览可先展示缩略图再按需加载原图。
- **表情/贴纸/Markdown**：消息 type 区分，content 或 metadata 存 id/结构化内容；渲染端按 type 走不同组件（emoji 字体、贴纸图、md 渲染器）。
- **卡片/链接预览**：链接类消息可存 url + 服务端抓取的 title/desc/icon；或前端 meta 解析；小程序卡片为固定 schema，按 type 渲染。

### 6.3 实现要点

- 上传：接口设计为 multipart 或分片 uploadId；返回 url 写入 message.content 或 metadata；客户端发消息时先上传再发或「先发占位后补 url」看产品。
- 富媒体：消息协议里 type + content + metadata 扩展；前端根据 type 映射到不同 UI 组件。

### 6.4 面试官可能问

- **Q：图片/视频为什么要先上传再发消息？**  
  A：消息体里只存 URL，保证消息可漫游、多端一致；若先发消息再上传，失败时消息已存在但无内容，要额外做「补传」与状态机。
- **Q：断点续传怎么实现？**  
  A：分片上传，每片带 index；服务端记录已传分片，客户端续传时只传未完成片；合并后生成最终文件 url。
- **Q：链接预览安全怎么保证？**  
  A：服务端抓取时限制域名、超时、大小；不做前端任意 URL 的 iframe，防钓鱼；可对预览图做内容安全检测。

## 7. 安全与反垃圾

### 7.1 一句话

**内容安全**：文本/图片/音视频走审核服务（敏感词、违禁图）；**防篡改/防刷**：签名校验、频率限制、设备/行为风控。

### 7.2 技术原理

- **内容审核**：消息落库前或异步送审；文本敏感词过滤/替换，图片/音频走云厂商审核接口；不通过则拒发或仅自己可见并提示。
- **防撤回/防篡改**：关键消息可做 hash 或签名存证；展示层不信任客户端下发的「已撤回」除非服务端下发；编辑记录可留痕。
- **防刷屏/防攻击**：单用户/单会话 QPS 限流、单日条数上限；验证码/设备指纹防机器人；敏感操作（如建群、拉人）可二次校验。

### 7.3 实现要点

- 服务端：发消息 pipeline 中接入审核与限流；存储可留 contentHash；限流用 Redis 计数。
- 客户端：被拒发时展示明确原因；不尝试绕过限流（否则封号风险）。

### 7.4 面试官可能问

- **Q：审核延迟高怎么办？**  
  A：先发后审：消息先落库并推送，审核异步；不通过再撤回或置为「违规」并通知发送者。
- **Q：端到端加密和内容审核矛盾吗？**  
  A：会矛盾：端到端加密后服务端看不到明文，无法做内容审核；合规场景通常用「服务端可解密」或仅对部分字段加密，审核后再加密存储。

## 8. 推送体系

### 8.1 一句话

**离线推送**：用户离线时消息先落库，通过 APNS/FCM/HMS 发推送；**收敛与互斥**：合并多条为一条通知、在线则不推或仅 badge 更新。

### 8.2 技术原理

- **离线推送**：长连断开或推送服务检测到设备离线时，新消息写入待推送队列；按设备 token 调厂商通道发 push；payload 可只带 conversationId、messageId、摘要，点击再拉详情。
- **推送收敛**：同一会话 N 条未读可合并为「张三发来 3 条消息」；可按时窗（如 5 分钟内）或条数阈值合并。
- **与在线互斥**：若该用户当前有长连接在线且消息已通过长连下发，则不再发离线 push，仅更新角标或静默同步。

### 8.3 实现要点

- 服务端：维护用户-设备 token 与在线状态；消息投递时判断在线则走长连，离线则入 push 队列；推送服务消费队列、调厂商 API、做收敛与去重。
- 客户端：上报 token、处理点击跳转到对应会话/消息。

### 8.4 面试官可能问

- **Q：推送送达率怎么保障？**  
  A：厂商通道本身有送达率；可做多厂商冗余（如国内 HMS + FCM 降级）；失败重试、token 失效时让客户端重新上报。
- **Q：如何避免「已读还推」？**  
  A：推送前查该会话在该设备是否已读（lastReadSeq）；若已读则不推或只更新角标。

## 9. 群聊高级功能

### 9.1 一句话

**群能力**：公告、禁言、@ 与权限在服务端维护；**万人群**：扩散用分层、分级推送或只推摘要，减少扇出与存储放大。

### 9.2 技术原理

- **群公告/禁言**：群元数据表存公告内容、禁言名单/全员禁言；发消息时校验是否被禁言；公告变更推送全群。
- **@所有人 / @某人**：消息 metadata 带 atUserIds；@所有人 仅群主/管理员可用，服务端校验；展示时高亮并可选推送被 @ 的人。
- **权限**：群主/管理员/普通成员；踢人、改群信息、禁言、@所有人 等按角色校验。
- **超大群优化**：消息不按「每人写一条」扩散，而是写一条群消息，在线成员按会话订阅拉取；离线用「未读计数 + 摘要」推送而非全量；历史可按需拉取或只保留最近 N 条。

### 9.3 实现要点

- 服务端：群表、群成员表（含 role）；发消息时查禁言、@ 权限；大群可单独存储与扩散策略（如只存一条，读扩散）。
- 客户端：展示 @ 高亮、公告入口、禁言提示。

### 9.4 面试官可能问

- **Q：万人群发一条消息，存储与推送量级？**  
  A：存：一条消息一条记录，不按人复制。推：在线用户按会话订阅收一条；离线可按「未读数+最后一条摘要」合并推送，不全量推。
- **Q：@所有人 怎么不刷屏？**  
  A：仅一条消息里带 atAll 标记；推送时可对「被 @ 的用户」做一次提醒，不重复 N 次。

## 10. 性能与体验优化（架构师重点）

### 10.1 一句话

**长连保活**：自适应心跳、省电策略；**消息与列表**：合并请求、虚拟列表、懒加载；**弱网**：重连退避、压缩、二进制协议。

### 10.2 技术原理

- **智能心跳**：根据网络类型与上次活动调整心跳间隔（如 WiFi 30s、4G 60s）；有消息收发可顺带充当心跳，减少空包。
- **合并请求**：多条消息可合并为一个包发送（batch）；ACK 可批量回；拉历史可一次拉多会话的增量。
- **虚拟列表**：只渲染可视区 + overscan，消息列表用 react-virtuoso 等，减少 DOM 与重排。
- **弱网/断网重连**：断线后指数退避重连（1s、2s、4s… 上限如 60s）；重连后拉离线消息、同步已读与状态。
- **流量压缩**：协议用 Protobuf 等二进制；HTTP 层 GZIP；长连包体尽量小，避免重复字段。

### 10.3 实现要点

- 客户端：心跳 timer 随网络与活动调整；消息队列合并后发送；列表用虚拟滚动；断线检测 + 退避重连。
- 服务端：支持批量 ACK、批量拉取；推送包体精简；可做连接层压缩。

### 10.4 面试官可能问

- **Q：心跳间隔设多少合适？**  
  A：平衡及时性与流量/电量：一般 30–60s；移动端可更长或按网络自适应；有数据可捎带心跳。
- **Q：虚拟列表在 IM 里要注意什么？**  
  A：新消息到底部要 followOutput；滚动到历史要 loadMore；item 高度不固定时需测量或估算，避免跳动。
- **Q：为什么用 Protobuf 而不是 JSON？**  
  A：体积小、解析快，高 QPS 与弱网下省带宽与 CPU；代价是调试与兼容性，需版本管理。
