# 前端状态管理方案对比

本文档对比 React 生态中主流的几种状态管理方案，包括优缺点、使用场景及选型建议。

---

## 一、方案概览

| 方案 | 类型 | 学习成本 | 包体积 | 适用规模 | 维护方 |
|------|------|----------|--------|----------|--------|
| React Context | 内置 | 低 | 0 | 小 | React 官方 |
| Redux / RTK | 单一 store | 中～高 | ~12KB (RTK) | 大 | Redux 团队 |
| Zustand | 单一/多 store | 低 | ~1KB | 小～大 | 社区 |
| Jotai | 原子化 | 低 | ~3KB | 小～中 | 社区 |
| Recoil | 原子化 | 中 | ~14KB | 中～大 | Meta |
| MobX | 响应式 | 中 | ~16KB | 中～大 | 社区 |

---

## 二、各方案详解

### 2.1 React Context + useReducer

**思路**：用 Context 传递状态，用 `useReducer` 管理更新逻辑。

```tsx
const [state, dispatch] = useReducer(reducer, initialState);
<Context.Provider value={{ state, dispatch }}>{children}</Context.Provider>
```

#### 优点
- **零依赖**：React 内置，无需额外包
- **概念简单**：Provider / Consumer，易理解
- **与 React 集成好**：符合 React 数据流习惯

#### 缺点
- **重渲染问题**：Context 值变化时，所有消费该 Context 的组件都会重渲染，难以细粒度订阅
- **Provider 嵌套**：多 Context 时会出现深层嵌套
- **不能脱离 React 使用**：异步回调、工具函数中无法直接 `dispatch`
- **缺少 DevTools**：没有官方时间旅行、状态快照等

#### 适用场景
- 简单全局配置（主题、语言、用户信息）
- 小规模应用，状态变更不频繁
- 对依赖零要求、不引入第三方库的项目

---

### 2.2 Redux / Redux Toolkit (RTK)

**思路**：单一 store，通过 dispatch(action) 触发 reducer 更新，单向数据流。

```tsx
const store = configureStore({ reducer: rootReducer });
<Provider store={store}><App /></Provider>
// 组件中
const count = useSelector(state => state.counter.value);
dispatch(increment());
```

#### 优点
- **可预测性强**：纯函数 reducer，便于测试和调试
- **生态成熟**：中间件（thunk、saga）、DevTools、持久化等
- **时间旅行**：Redux DevTools 支持回溯、重放
- **规范统一**：适合大团队、多人协作
- **RTK 降低复杂度**：`createSlice`、`createAsyncThunk` 减少样板代码

#### 缺点
- **学习成本高**：action、reducer、middleware 等概念
- **样板代码多**：即便有 RTK，仍比 Zustand 等繁重
- **包体积较大**：redux + react-redux + RTK 约 12KB+
- **异步需额外处理**：需 thunk 或 saga，不是「直接写 async」

#### 适用场景
- 中大型应用，状态结构复杂
- 需要严格审计、可追溯的数据流
- 多人协作、需要统一规范
- 需要强 DevTools、中间件扩展（日志、持久化、埋点等）

---

### 2.3 Zustand

**思路**：轻量 store，`create` 创建，`useStore` 订阅，支持在 React 外使用 `getState` / `setState`。

```tsx
const useStore = create((set) => ({
  count: 0,
  increment: () => set((s) => ({ count: s.count + 1 })),
}));
// 组件中
const count = useStore((s) => s.count);
const increment = useStore((s) => s.increment);
```

#### 优点
- **极简 API**：create + set + get，上手快
- **细粒度订阅**：selector 只在返回值变化时触发重渲染
- **可脱离 React**：`getState()`、`setState()` 可在任意位置调用
- **体积小**：约 1KB gzip
- **无 Provider**：无需包裹根组件
- **支持中间件**：persist、devtools、immer 等

#### 缺点
- **无强制规范**：写法自由，团队需自行约定
- **生态小于 Redux**：中间件、示例相对少
- **大项目需自行拆分**：多 store 或模块化需设计

#### 适用场景
- 中小型应用、MVP、内部工具
- 对体积敏感、追求简洁的项目
- 需要在非 React 环境读写状态（如 WebSocket 回调、SDK）
- 本 IM 项目即采用 Zustand

---

### 2.4 Jotai

**思路**：原子化状态，每个 atom 是独立单元，通过 `useAtom` 组合使用。

```tsx
const countAtom = atom(0);
const doubledAtom = atom((get) => get(countAtom) * 2);
// 组件中
const [count, setCount] = useAtom(countAtom);
const doubled = useAtomValue(doubledAtom);
```

#### 优点
- **按需渲染**：只订阅用到的 atom，自动细粒度更新
- **派生状态自然**：通过 `atom((get) => ...)` 表达衍生数据
- **代码分割友好**：atom 可 lazy 初始化
- **与 Suspense 配合好**：支持异步 atom
- **类型友好**：TypeScript 体验好

#### 缺点
- **概念略抽象**：atom、derived atom 需要理解
- **全局状态不如 Zustand 直观**：大量 atom 时需规划结构
- **体积略大**：约 3KB

#### 适用场景
- 希望细粒度、按需更新的应用
- 大量派生状态、依赖图较复杂
- 需要 Suspense、代码分割的项目

---

### 2.5 Recoil

**思路**：原子化 + selector，类似 Jotai，但 API 和心智模型不同。

```tsx
const countState = atom({ key: 'count', default: 0 });
const doubledSelector = selector({
  key: 'doubled',
  get: ({ get }) => get(countState) * 2,
});
// 组件中
const [count, setCount] = useRecoilState(countState);
const doubled = useRecoilValue(doubledSelector);
```

#### 优点
- **Meta 官方**：与 React 团队关系近
- **细粒度更新**：atom 级订阅
- **异步 selector**：内置支持 async get
- **DevTools**：有官方扩展

#### 缺点
- **实验性**：文档注明 Experimental
- **体积大**：约 14KB
- **学习曲线**：atom、selector、key 等概念
- **生态一般**：社区活跃度不如 Zustand / Jotai

#### 适用场景
- 需要原子化 + 异步派生状态
- 对 Meta 技术栈有偏好
- 中大型应用，可接受实验性 API

---

### 2.6 MobX

**思路**：响应式，用 observable 包装状态，自动追踪依赖并更新视图。

```tsx
class Store {
  count = 0;
  increment() { this.count++; }
}
const store = makeAutoObservable(new Store());
// 组件中
const count = observer(() => store.count);
```

#### 优点
- **写法贴近 OOP**：适合习惯 class、命令式的开发者
- **自动依赖追踪**：无需手动 selector
- **性能好**：精确更新，只重渲染依赖变化的组件
- **异步友好**：`runInAction`、`flow` 等

#### 缺点
- **隐式魔法**：依赖 Proxy/装饰器，调试需理解原理
- **包体积大**：约 16KB
- **与 React 范式不同**：mutable、观察者模式，和「不可变 + 单向数据流」有差异
- **SSR 需注意**：需配合 mobx-react-lite 等

#### 适用场景
- 复杂领域模型、富交互
- 团队熟悉 OOP、响应式编程
- 对「自动依赖追踪」有强需求

## 三、选型决策树

```
是否需要时间旅行 / 强审计？
├─ 是 → Redux (RTK)
└─ 否
    ├─ 是否强依赖 React 内置能力？
    │   ├─ 是、且状态极简 → Context + useReducer
    │   └─ 否
    ├─ 是否需要在非 React 环境读写状态？
    │   ├─ 是 → Zustand
    │   └─ 否
    ├─ 是否大量派生状态、希望细粒度按需更新？
    │   ├─ 是 → Jotai 或 Recoil
    │   └─ 否
    ├─ 是否偏好 OOP、响应式？
    │   ├─ 是 → MobX
    │   └─ 否
    └─ 默认推荐 → Zustand（平衡体积、简单、灵活性）
```

---

## 四、与本项目的关系

本 IM 项目选用 **Zustand**，主要考虑：

1. **WebSocket 回调**：需在非 React 路径（如 `client.on(SDKEvent.xxx)`）更新 UI，Zustand 的 `getState()` / `set()` 天然支持
2. **单 Store 足够**：会话级 IM，状态集中，无需 Redux 的复杂分层
3. **体积敏感**：Next.js 前端，希望尽量减少 JS 体积
4. **开发效率**：无 boilerplate，快速迭代

详细设计见 `技术方案.md` 及之前的「状态管理分析」讨论。

## 五、总结表

| 需求 | 推荐方案 |
|------|----------|
| 零依赖、极简 | Context + useReducer |
| 大型、规范、审计 | Redux (RTK) |
| 通用、轻量、灵活 | Zustand |
| 细粒度、派生多 | Jotai |
| 原子化 + 异步 | Recoil |
| OOP、响应式 | MobX |
