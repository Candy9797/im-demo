## 一、架构

### 面试官追问清单

1. **整体架构是什么？分层和职责？**
2. **为什么用 WebSocket 而不是 HTTP 轮询？**
3. **IMClient 和 WebSocketManager 的职责边界？为什么要分开？**
4. **Store 和 SDK 之间的数据流是怎样的？**

---

**Q：整体架构？**

分层架构。最上层是 Zustand + React 的 UI 层，`chatStore.ts` 用 `create(immer(...))` 创建，组件通过 `useChatStore`、`useShallow` 订阅。中间是自研 SDK（`src/sdk/`）：IMClient 负责会话管理和事件派发；MessageQueue 负责入站 enqueueIncoming/flushIncoming、出站 enqueueOutgoing/flushOutgoing；WebSocketManager 负责 connect、send、心跳、重连。底层是 WebSocket（ws://host:3001/ws）和 IndexedDB（库名 web3-im）。Store 在 initialize 时创建 IMClient 并订阅 SDKEvent（如 MESSAGE_BATCH_RECEIVED、PHASE_CHANGED），收到后 set 更新，IMClient 不直接操作 UI。

**Q：为什么用 WebSocket？**

WebSocket 是全双工长连接，建立一次连接后双向通信，延迟低；服务端可主动推送，适合实时消息。HTTP 轮询需要客户端不断发起请求，延迟取决于轮询间隔，且大量无效请求浪费带宽和服务器资源。IM 对实时性要求高，WebSocket 是更合适的选择。

**Q：IMClient 和 WebSocketManager 职责边界？**

WebSocketManager（`WebSocketManager.ts`）只负责传输层：建立/关闭 WebSocket、收发 JSON 帧（`{ type, seq, timestamp, payload }`）、心跳（heartbeatInterval 30000ms 发 HEARTBEAT_PING）、断线重连（reconnectAttempts 5、reconnectInterval 1000、指数退避）。不解析业务、不处理消息语义。IMClient（`IMClient.ts`）负责业务层：sendMessage/createMessage、handleOutgoingBatch、frame_in 分发、message_ack 匹配、loadHistory、sync、requestHumanAgent 等，内部 `this.wsManager = new WebSocketManager(config)`，通过 `wsManager.send(type, payload)` 和 `wsManager.on(type, cb)` 交互。

**细化（可展开）**：

- **为什么分层**：传输层（WS）与业务层（消息语义、ACK、同步）解耦，WS 可替换为其他传输，业务逻辑复用。
- **数据流**：UI → chatStore.sendMessage → IMClient.sendMessage → MessageQueue → WebSocketManager.send；反向：ws.onmessage → IMClient.frame_in → emit(SDKEvent) → chatStore.set → 组件重绘。

## 二、消息可靠性

### 面试官追问清单

1. **如何保证消息不丢？发送侧和接收侧分别怎么做？**
2. **如何保证消息不重？有哪些防护层？**
3. **client_msg_id 和 server_msg_id 为何分开？何时替换？**
4. **消息顺序如何保证？seq_id 的作用？**

**Q：如何保证消息不丢？**

发送侧：乐观更新，用户发消息后立即展示，不等 ACK；ACK 失败时 MessageQueue 会重试，指数退避最多 3 次。接收侧：每条消息收到后写入 IndexedDB，刷新或断网再打开也能从本地恢复。断线恢复：重连成功后发 SYNC 帧，payload 为 `{ afterSeqId }`，服务端返回该 seq 之后的消息，与本地合并，保证断线期间的消息不丢。

**Q：如何保证消息不重？**

多层防护。MessageQueue 的 seenIds（Map<id, timestamp>）配合 deduplicationWindow 5000ms，cleanupDedup 清理过期条目，enqueueIncoming 时 isDuplicate 检查。Store 在 MESSAGE_BATCH_RECEIVED、HISTORY_LOADED 时 `const ids = new Set(state.messages.map(m => m.id))`，过滤 `!ids.has(m.id)`。IMClient 的 frame_in 有 `conversation.messages.some(m => m.id === msg.id)` 检查。IndexedDB 的 ObjectStore（STORE_NAME: "messages"）keyPath 为 "id"，put 覆盖同 id。网络重试、服务端重发、多端同步都可能造成重复，多层过滤保证最终展示无重复。

**Q：client_msg_id 和 server_msg_id 为何分开？**

用户发消息需要立刻展示（乐观更新），此时服务端还未落库，没有 server_msg_id。客户端在 IMClient.createMessage 中生成 client_msg_id，格式 `msg-${Date.now()}-${Math.random().toString(36).slice(2, 8)}`，用于临时标识和 ACK 匹配。服务端 ws-handler 落库时生成 server_msg_id，ACK 帧（FrameType.MESSAGE_ACK）payload 携带 `{ clientMsgId, serverMsgId, seqId }`。IMClient 收到后 `conversation.messages.find(m => m.id === clientMsgId)`，替换 `msg.id = serverMsgId`、`msg.status = SENT`。之后 markAsRead、addReaction 等都用 server_msg_id。

**Q：消息顺序如何保证？**

服务端为每条消息分配单调递增的 seq_id。历史拉取按 beforeSeqId 分页，同步按 afterSeqId 补拉，都是按 seq 范围查询。客户端合并时按 seq_id 或 timestamp 排序，展示时统一有序。服务端写入在同一会话内保证 seq 递增，客户端拉取和合并都依赖 seq 来维持顺序。

**细化（可展开）**：

- **不丢**：发送侧乐观更新+ACK 重试 3 次；接收侧 IndexedDB 落库；断线 SYNC 按 afterSeqId 补拉。
- **不重**：MessageQueue seenIds + deduplicationWindow；Store Set 过滤；IMClient frame_in 检查；IndexedDB keyPath id 覆盖。
- **双 id**：createMessage 生成 client_msg_id → 乐观展示；ACK 返回 server_msg_id → 替换；markAsRead/addReaction 支持双 id 查找。

## 三、连接与重连

### 面试官追问清单

1. **断线重连的完整流程？pause/resume 在什么时候？**
2. **为什么用指数退避？惊群效应是什么？**
3. **心跳机制的作用？间隔怎么选？**

**Q：断线重连流程？**

WebSocketManager 在 ws.onclose / ws.onerror 时进入 handleDisconnect。若 `reconnectCount < reconnectAttempts`（默认 5），则 scheduleReconnect：`delay = min(1000 * 2^reconnectCount + Math.random()*1000, 30000)`，setState(RECONNECTING)、emit(RECONNECTING)，setTimeout 后再次 connect。IMClient 在连接前/断线时调用 messageQueue.pause()，auth_ok 回调里启动 MessageQueue 后、或收到 sync_response 后 resume()。重连成功后发 FrameType.SYNC，payload `{ afterSeqId }`，服务端返回 sync_response，客户端合并、去重、按 seq 排序。

**Q：为什么用指数退避？**

如果所有客户端断线后同时重连，会造成服务端瞬时流量激增（惊群效应），可能拖垮服务。指数退避让每次重试间隔逐渐变长，加上随机抖动，不同客户端的重连时间错开，分散压力。公式 `base * 2^n + jitter`，n 为已重试次数，上限 30 秒。

**Q：心跳机制的作用？**

WebSocket 长时间无数据时，中间网络（NAT、防火墙、代理）可能回收连接，但两端不一定立即感知。心跳每 30 秒发 HEARTBEAT_PING，服务端回 HEARTBEAT_PONG，起到保活作用；若连接已死，onclose 会触发，从而启动重连。心跳间隔需在「及时发现问题」和「不过度占用带宽」之间权衡。

**细化（可展开）**：

- **断线流程**：ws.onclose → handleDisconnect → pause MessageQueue → scheduleReconnect（指数退避）→ connect → auth_ok → resume → 发 SYNC → 合并 sync_response。
- **指数退避公式**：`min(1000 * 2^n + jitter, 30000)`，n 为已重试次数。
- **惊群效应**：大量客户端同时断线后同时重连，服务端瞬时压力激增，可能拖垮；退避+随机让重连时间错开。

## 四、高频优化

### 面试官追问清单

1. **高 QPS 场景下，入站、出站、IndexedDB、渲染分别怎么优化？**
2. **批处理间隔为什么选 50ms？**
3. **MessageQueue 的 pause/resume 有什么用？什么时候 pause？**

**Q：高 QPS 场景如何优化？**

入站：MessageQueue 的 enqueueIncoming 入队，flushTimer 每 flushInterval 50ms（IMClient 构造时传入）执行 flush，flushIncoming 每次取 batchSize 30 条，调用 onFlushIncoming → handleIncomingBatch → emit(MESSAGE_BATCH_RECEIVED)，Store 单次 set 合并。IndexedDB：saveMessage 用 writeBuffer + scheduleFlush（WRITE_DEBOUNCE_MS 80）、MAX_BUFFER_SIZE 50 即 flushBuffer；saveMessages 用于 auth_ok、sync 大批量。渲染：MessageList 用 Virtuoso，OVERSCAN 约 5，computeItemKey 用 message.id。服务端 ws-handler：checkRateLimit 滑动窗口 RATE_LIMIT_MSGS_PER_SEC 20；insertMessages 批量事务。

**Q：为什么批处理用 50ms？**

间隔太短（如 10ms）批处理效果有限，setState 仍然很频繁。间隔太长（如 200ms）用户会感觉消息有延迟、不够实时。50–100ms 是常见折中：每秒最多约 20 次 flush，能显著减少渲染，同时延迟在可接受范围内。具体值可根据业务调参。

**Q：MessageQueue 的 pause/resume 有什么用？**

断线重连时，如果继续 flush 入队消息，可能把基于不完整状态或重复推送的数据推到 Store，造成乱序或重复。重连期间 pause，等连接恢复、SYNC 补拉完成、状态稳定后再 resume，保证数据一致性。pause 期间消息仍会入队，只是不 flush 到上层。

**细化（可展开）**：

- **入站**：enqueueIncoming → flush 50ms/batchSize 30 → MESSAGE_BATCH_RECEIVED → 单次 set 合并。
- **IndexedDB**：writeBuffer + 80ms 防抖 + 50 条即 flush。
- **渲染**：Virtuoso 虚拟列表，OVERSCAN 5，computeItemKey 用 id。
- **50ms 取舍**：太短批处理效果差，太长用户感觉延迟；50–100ms 是常见折中。

## 五、离线与持久化

### 面试官追问清单

1. **IndexedDB 的写入策略？单条和批量分别怎么处理？**
2. **为什么 IndexedDB 要防抖？不防抖会怎样？**
3. **为何用 IndexedDB 存消息？localStorage 不行吗？**

**Q：IndexedDB 的写入策略？**

单条：saveMessage 将消息 `writeBuffer.set(msg.id, msg)`，若 size >= 50 则 clearTimeout、await flushBuffer；否则 scheduleFlush（80ms 后 flushBuffer）。flushBuffer 将 `Array.from(writeBuffer.values())` 交给 doWrite，put 到 ObjectStore，writeBuffer.clear()。批量：saveMessages 先 flush 未写 buffer（若有 flushTimer 则先 flush），再 doWrite(messages)。getMessages 前若 writeBuffer.size > 0 且 flushTimer 存在，先 flush 再 openDB、getAll、按 conversationId 过滤、按 timestamp 排序。

**Q：为什么 IndexedDB 要防抖？**

IndexedDB 的 put 是异步 I/O，频繁写入会占用主线程、影响渲染和交互。高频场景（如每秒多条 ACK）下，防抖和缓冲能把多次写入合并为一次事务，显著降低 I/O 次数，同时 80ms 和 50 条上限保证数据不会积压太久。

**Q：为何用 IndexedDB 存消息？**

为本地持久化。刷新页面、关闭标签后，消息仍在浏览器本地，下次打开可恢复。断网或弱网时，可先从 IndexedDB 读取已有会话，再等 sync 补拉。冷启动时，auth_ok 未到达前也能展示历史，减少白屏。弱网是典型场景之一，但刷新、关标签在正常网络下也会发生，不只是弱网需求。

**细化（可展开）**：

- **localStorage vs IndexedDB**：localStorage 同步、约 5MB、存字符串；IndexedDB 异步、容量大、支持索引和事务，适合大量结构化消息。
- **单条**：saveMessage → writeBuffer.set → size>=50 或 80ms 后 flushBuffer。
- **批量**：saveMessages 先 flush 未写 buffer，再 doWrite(messages)。
- **读前一致性**：getMessages 前若 writeBuffer 有数据，先 flush 再读，保证读到最新。

## 六、Zustand + Immer

### 面试官追问清单

1. **Immer 会冻结是什么意思？有什么影响？**
2. **为什么 push 消息时要拷贝 { ...m }？不拷贝会怎样？**
3. **chatStore 的数据流是怎样的？事件驱动还是轮询？**

**Q：Immer 会冻结是什么意思？**

Immer 的 produce 完成后，会对生成的新 state 调用 Object.freeze，使其不可变。这样外部无法直接修改 state，只能通过 set 更新，有利于可预测性和调试。freeze 是递归的，嵌套对象也会被冻结。

**Q：为何 push 消息时要拷贝 { ...m }？**

事件回调收到的 message 来自 IMClient，若直接 `state.messages.push(message)`，Store 和 IMClient 会共享同一对象引用。Immer 会 freeze 新 state，包括该 message。之后 IMClient 在处理 ACK 时需要修改这条消息（如替换 id、更新 status），对象已被 freeze，修改会静默失败或报错。拷贝 `{ ...message }` 后，Store 持有独立副本，freeze 不影响 IMClient 侧对象。

**Q：chatStore 的数据流？**

Zustand + Immer：`create<ChatState>()(immer((set, get) => ({ ... })))`，set 内可写式更新 draft（如 state.messages.push）。事件驱动：initialize 时 `client.on(SDKEvent.MESSAGE_BATCH_RECEIVED, ...)` 等，收到后 set 或 set(state => { ... })。流程：UI 调 sendMessage → get().client.sendMessage → IMClient 乐观更新并发帧；服务端回 message → IMClient emit → Store set → 订阅该 state 的组件重绘。组件用 useShallow 做浅比较，避免整 store 变化触发重渲染。scrollToInputRequest 用时间戳作信号，replyToMessage 时 set，InputArea 监听后滚动并聚焦。

**细化（可展开）**：

- **冻结**：produce 完成后 Object.freeze 新 state，递归；外部无法直接修改，只能通过 set，利于可预测性。
- **拷贝原因**：IMClient 和 Store 共享同一引用时，Immer freeze 后 IMClient 无法再改 message（如 ACK 时替换 id），会静默失败。
- **useShallow**：避免 `{ messages }` 引用变化导致整列表重渲染，只比较浅层字段。

## 七、实现难点

### 面试官追问清单

1. **EmojiPicker 被父级 overflow 裁剪怎么解决？**
2. **Virtuoso 父容器高度为 0 怎么解决？**
3. **引用回复的乐观更新如何实现？**
4. **消息反应的乐观更新如何实现？**

**Q：弹层选择器（EmojiPicker）被父级 overflow 裁剪怎么办？**

InputArea、MessageReactions 等用 `createPortal(children, document.body)` 将 EmojiPicker、StickerPicker 渲染到 body，脱离 MessageList 的 overflow。用 `position: fixed` + `anchorRef.current.getBoundingClientRect()` 计算 top/left，useLayoutEffect 或 resize/scroll 监听时 updatePickerPosition。MessageReactions 的 picker 在气泡内，气泡随列表滚动，需监听 scroll 并更新坐标。

**Q：Virtuoso 父容器高度为 0 怎么解决？**

flex 子元素默认 min-height: auto 会阻止收缩。MessageList 的父链：.chat-window-inner → flex:1 min-height:0，.message-list-wrap → flex:1 min-height:0 position:relative，.message-list-inner → position:absolute inset:0，Virtuoso 作为其子元素获得确定高度。history 页同理，.history-main、.history-list-wrap 设 min-height:0，.history-message-list 用 absolute inset:0。

**Q：引用回复的乐观更新如何实现？**

发送时立即构造带 `metadata.quote` 的消息（包含被引用消息的 messageId、senderName、content 等）推入列表，用户立刻看到引用块。ACK 返回时携带 clientMsgId 和 serverMsgId，用 clientMsgId 找到乐观消息，替换 id 并更新 status。引用目标可能只有 client_msg_id（如刚发的消息），展示时需兼容两种 id。

**Q：消息反应的乐观更新如何实现？**

addReaction 在 Store 中直接修改对应消息的 `metadata.reactions`（按 emoji → userId[] 存储），UI 立刻显示反应。然后调用 client.addReaction 同步服务端，服务端 getMessage 支持按 client_msg_id 查找，更新后广播 REACTION_UPDATE。同一用户对同一 emoji 只记录一次，去重逻辑在服务端和客户端都有。

**细化（可展开）**：

- **弹层**：createPortal 渲染到 body，position: fixed + getBoundingClientRect 计算位置，监听 scroll/resize 更新。
- **Virtuoso 高度**：flex 子元素 min-height: auto 会阻止收缩；父链设 `min-height: 0`，再用 `position: absolute; inset: 0` 给 Virtuoso 确定高度。
- **引用回复**：发送时 metadata.quote 含被引用消息 id/name/content；ACK 时用 clientMsgId 找乐观消息替换 id。
- **反应**：本地先改 metadata.reactions，再 client.addReaction；服务端双 id 查找、广播 REACTION_UPDATE。

## 八、AI 问答页

### 面试官追问清单

1. **AI 页如何实现流式？从 API 到前端渲染的完整流程？**
2. **SSE 和 WebSocket 的区别？为什么 AI 用 SSE？**
3. **如何对接大模型（如 OpenAI）？**

**Q：AI 页如何实现流式？**

POST /api/ai/chat，body `{ message: string }`。API（`src/app/api/ai/chat/route.ts`）返回 `Content-Type: text/event-stream`，ReadableStream 中 `controller.enqueue(encoder.encode("data: "+JSON.stringify({content})+'\n\n'))`。前端 `fetch` → `res.body.getReader()` → 循环 `reader.read()`，decoder.decode(value)，buffer 按 `\n` 分割解析，`line.startsWith("data: ")` 则 JSON.parse 取 content，`setMessages(prev => prev.map(m => m.id===assistantId ? {...m, content: m.content+chunk} : m))`。流结束发 `[DONE]`，客户端 `if (chunk==='[DONE]') continue`。AIContent 用 ReactMarkdown 渲染，AICodeBlock、AIImage 为自定义组件。

**Q：SSE 和 WebSocket 的区别？**

SSE 基于 HTTP，单向（服务端→客户端），自动重连，数据为文本格式。WebSocket 是独立协议，全双工，需自己实现重连。AI 流式场景只需服务端推送文本，不需要客户端频繁上行，SSE 更简单，且复用 HTTP 基础设施。若需客户端频繁发送或二进制数据，用 WebSocket 更合适。

**Q：如何对接大模型？**

在 route.ts 中替换 mock 逻辑，调用 OpenAI / Claude 等 streaming API。例如 OpenAI：`openai.chat.completions.create({ stream: true, ... })`，for await 遍历 stream，取 `chunk.choices[0]?.delta?.content`，转为 `data: { content }` 格式输出。前端解析逻辑不变，只需保证输出格式一致。可加对话历史，将 messages 数组传给 API 实现多轮对话。

**细化（可展开）**：

- **流式流程**：API 返回 ReadableStream → res.body.getReader() → 循环 read → 按 `\n` 解析 data: 行 → JSON.parse 取 content → setMessages 追加到 assistant 消息。
- **SSE vs WS**：SSE 基于 HTTP、单向、自动重连、文本；WS 全双工、需自实现重连。AI 只需服务端推送文本，SSE 更简单。
- **对接大模型**：openai.chat.completions.create({ stream: true }) → for await 取 delta.content → 输出 data: { content }。

## 九、认证与扩展

### 面试官追问清单

1. **认证方式有哪些？SIWE、访客、WebSocket 鉴权怎么做的？**
2. **Bot 和 Agent 会话的区别？转人工流程？**
3. **多实例部署要考虑什么？**
4. **为什么用 SQLite？**

**Q：认证方式？**

SIWE：`lib/siwe.ts` 的 getNonce、connectAndSign、verifyAndGetToken；钱包 signMessage 后 POST /api/auth/verify 换 JWT。访客：GET /api/auth/demo（Next.js route 代理到 3001），返回 { token, userId, address }。WebSocket：URL `ws://host:3001/ws?token=xxx` 或 `&fresh=1`（新建会话不拉历史）；ws-handler handleConnection 中 verifyToken(token)，失败则 send ERROR、ws.close()。auth.ts 的 verifyToken 用 jwt.verify。

**Q：Bot 和 Agent 会话的区别？**

Bot（conv-bot-*）：用户与 Smart Assistant，phase=BOT。Agent（conv-agent-*）：转人工后，phase=AGENT，parent_conv_id 关联 Bot。转人工流程：客户端 requestHumanAgent() 发 REQUEST_AGENT；服务端 handleRequestAgent 创建 Agent 会话、phase=QUEUING、每 2 秒 position-1 发 QUEUE_STATUS；position 到 0 时 assignAgent（固定 Agent #1024）、发 SESSION_SWITCHED；客户端收到后 set conversationId、messages、phase=AGENT。QueueBanner 在 QUEUING 时展示排队位置和 estimatedWait。

**Q：多实例部署要考虑什么？**

连接和会话需在实例间共享：用 Redis 存 connsByUser、限流计数等。消息路由需能跨实例投递：用户可能连在 A 实例，消息从 B 实例产生，需转发。可选消息队列（Kafka/RabbitMQ）做异步处理，解耦写入与推送。

**Q：为何用 SQLite？**

当前单机部署，SQLite 无额外依赖、配置简单，WAL 模式下读写并发足够。数据文件单文件，备份方便。若需多写、分布式、高可用，再考虑 PostgreSQL 等。

**细化（可展开）**：

- **SIWE**：getNonce → 钱包 signMessage → POST /api/auth/verify 换 JWT；WebSocket URL 带 token，handleConnection 中 verifyToken。
- **Bot/Agent**：Bot phase=BOT；转人工 REQUEST_AGENT → 创建 Agent 会话、phase=QUEUING → 排队 position-- → assignAgent → SESSION_SWITCHED。
- **多实例**：Redis 存 connsByUser、限流；消息路由需跨实例；可选 Kafka 解耦写入与推送。

## 十、深挖实现细节

### 面试官追问清单

1. **完整消息发送链路（从用户点击发送到 UI 更新）？**
2. **auth_ok 有哪些分支逻辑？什么时候从 IndexedDB 拉？**
3. **MessageQueue 出队满时怎么办？发送失败怎么重试？**
4. **服务端 getMessage 如何支持双 id 查找？**
5. **SEND_MESSAGE 服务端有哪些处理分支（bot/agent/queuing）？**
6. **IndexedDB 读前如何保证一致性？**

### 10.1 完整消息发送链路

1. 用户输入 → sendMessage → `client.sendMessage(content)`  
2. IMClient：`createMessage` 生成 id → `conversation.messages.push` → `messageQueue.enqueueOutgoing` → `emit(MESSAGE_SENT, message)`  
3. chatStore 收到 MESSAGE_SENT → `set(state => state.messages.push({...message}))`，UI 展示  
4. MessageQueue flush（50ms 定时）→ `handleOutgoingBatch` → 逐条 `wsManager.send(FrameType.SEND_MESSAGE, msg)`  
5. 服务端 handleFrame：checkRateLimit → `db.nextSeqId` → `db.insertMessages`（用户消息 + Bot/Agent 回复同一事务）→ `sendToUser(MESSAGE_ACK)`、`sendToUser(MESSAGE)`  
6. 客户端 message_ack：`conversation.messages.findIndex(m => m.id === clientMsgId)` → 替换 id、status → emit MESSAGE_STATUS_UPDATE → saveMessage 写 IndexedDB  
7. Store 收到 MESSAGE_STATUS_UPDATE → `state.messages.find(x => x.id === message.id)` → `m.status = message.status`

### 10.2 auth_ok 分支逻辑

- 若 `serverMessages.length > 0`：直接覆盖 `conversation.messages`，saveMessages 落 IndexedDB，按 1/多条 emit MESSAGE_RECEIVED 或 MESSAGE_BATCH_RECEIVED  
- 若 `serverMessages.length === 0`：从 IndexedDB `getMessages(conversationId)` 拉本地，有则覆盖并 emit  
- 检查 welcome：`conversation.messages.some(m => m.type===SYSTEM && m.content?.includes("Welcome"))`，无则 unshift 欢迎消息并 emit  
- 最后 `messageQueue.start(onFlushOutgoing, onFlushIncoming)`，resolve(connect)

### 10.3 MessageQueue 出队满时策略

`enqueueOutgoing` 时若 `outgoing.length >= maxSize`：先找 `findIndex(p => p.message.status !== SENDING)`，删除 oldest 非发送中消息腾位；若全是 SENDING 则 `console.warn` 并 return false。发送失败重试：`flushOutgoing` 的 catch 中，`pending.attempts < retryAttempts` 则 `outgoing.unshift(pending)` 重入队头；否则 `pending.message.status = FAILED`，`incoming.push(pending.message)` 让 UI 展示失败态。

### 10.4 服务端 getMessage 双 id 查找

`db.getMessage(msgId, convId)`：先 `SELECT * FROM messages WHERE id = ?`；若无且 convId 存在，再 `SELECT * FROM messages WHERE conversation_id = ? AND client_msg_id = ?`。用于 MARK_READ、ADD_REACTION 等，消息可能尚未 ACK 只有 client_msg_id。

### 10.5 SEND_MESSAGE 服务端处理分支

- 限流失败：send ERROR rate_limit，break  
- phase=bot：贴纸直接 insertMessages + ACK 返回；否则 `getBotReply(content)`，为 null 则 insertMessages + ACK + `handleRequestAgent`（转人工）；有回复则用户+Bot 两条一起 insertMessages，ACK + MESSAGE  
- phase=agent：用户+Agent 回复两条 insertMessages，ACK + MESSAGE  
- phase=queuing：仅 insertMessages + ACK

### 10.6 SQL 与事务

- `getMessagesAfter`: `WHERE conversation_id=? AND seq_id>? ORDER BY seq_id ASC`  
- `getMessagesBefore`: `WHERE conversation_id=? AND seq_id<? ORDER BY seq_id DESC LIMIT ?`，beforeSeqId<=0 返回 []  
- `insertMessages` 用 `db.transaction` 包一坨 `STMT_INSERT_MSG.run(...)`，单事务批量插入  
- `nextSeqId`: `SELECT MAX(seq_id) FROM messages WHERE conversation_id=?` + 1

### 10.7 连接管理细节

- `connsByUser`: `Map<userId, Map<connId, ConnEntry>>`，支持多设备；`wsToConn`: `WeakMap<ws, ConnEntry>`  
- `kickOthers` 时遍历该用户旧连接，send KICKED、close，并 clear  
- `ws.on("close")`：从 connsByUser 删除 entry，若该用户无连接则 delete userId、rateLimitMap.delete、broadcastPresence  
- 转人工时 `for (e of connsByUser.get(userId)?.values() ?? []) { e.convId = agentConvId }`，切换所有连接会话

### 10.8 MessageList 与 Virtuoso

- `showAvatarMap`：相邻消息同 senderType 且时间差 ≤60s 则不显示头像  
- `followOutput`：`(isAtBottom) => isAtBottom ? 'smooth' : false`，仅当在底部时新消息平滑滚动  
- `atTopStateChange`：`atTop && hasMoreHistory` 时调用 `loadMoreHistory`  
- `loadMoreHistory`：取 `messages.filter(m => m.seqId!=null||m.timestamp)` 的 `Math.min(...)` 作为 beforeSeqId，`client.loadHistory(minSeq)`，set loadingHistory true，1.5s 后兜底设 false  
- `scrollToInputRequest`：replyToMessage 时 set 时间戳，useEffect 监听到变化后 `virtuosoRef.scrollToIndex(messages.length-1)`

### 10.9 转人工与排队的服务端实现

- `handleRequestAgent`：createAgentConversation → 切换该用户所有连接的 convId → updateConversation phase=queuing、queue_position=3、queue_total=5 → send PHASE_CHANGE → `setInterval` 每 2 秒 pos--，updateConversation、send QUEUE_STATUS，pos<=0 时 clearInterval、assignAgent  
- `assignAgent`：updateConversation phase=agent、agent_id 等，插入欢迎消息，send SESSION_SWITCHED

### 10.10 IndexedDB 读前一致性

`getMessages` 中：若 `writeBuffer.size > 0 && flushTimer`，则 clearTimeout(flushTimer)、flushTimer=null、await flushBuffer，再 openDB、getAll、filter、sort。保证读到的数据包含尚未 flush 的 buffer。

### 10.11 WebSocketManager send 前置条件

`send` 时 `if (state !== CONNECTED || !this.ws) throw new Error("Not connected")`。MessageQueue 的 handleOutgoingBatch 逐条 send，若未连接会抛错，被 flushOutgoing 的 catch 捕获，触发重试逻辑。

### 10.12 其他细节点

- **nonce 过期**：createNonce 写入 expires_at = now + 5min；consumeNonce 时 `expires_at > Date.now()` 才有效，用后 DELETE  
- **REACTION_UPDATE payload**：`{ messageId, clientMsgId, reactions, message? }`，支持按 client_msg_id 匹配未 ACK 消息  
- **history_response 合并**：`[...newMsgs, ...conversation.messages].sort((a,b) => (a.seqId??a.timestamp) - (b.seqId??b.timestamp))`，新消息 prepend 后整体排序  
- **sync_response**：服务端 `getMessagesAfter(convId, afterSeqId ?? 0)`，客户端 filter 去重后 push，emit MESSAGE_RECEIVED/BATCH  
- **createIMClient 默认**：messageQueueSize 1000（createIMClient 传入），IMClient 构造 MessageQueue 时 `maxSize: config.messageQueueSize || 2000`
