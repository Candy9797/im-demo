## 三、架构设计

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────────────────────┐
│                         Next.js (port 3000)                           │
│  page.tsx → ChatWidget → ChatWindow → MessageList / InputArea        │
│                                    │                                 │
│                          Zustand chatStore (Immer)                    │
│                                    │                                 │
│  IMClient ── WebSocketManager ── MessageQueue (批处理/去重)           │
│       │              │                     │                          │
│       └──────────────┼─────────────────────┼→ chatPersistStorage     │
│                      │                     │   (IndexedDB 80ms 防抖)  │
└──────────────────────┼─────────────────────┼─────────────────────────┘
                       │ WebSocket ?token=   │
                       ▼                     │
┌──────────────────────┼─────────────────────┼─────────────────────────┐
│                 IM Server (port 3001)       │                          │
│  ws-handler ← auth / bot / upload           │                          │
│       │                     │                                          │
│       └─────────────────────┴→ SQLite (data/im.db, WAL 模式)           │
└──────────────────────────────────────────────────────────────────────┘
```

### 3.2 模块职责（体现分层思维）

| 模块 | 职责 | 依赖 |
|------|------|------|
| **IMClient** | 会话管理、消息收发、ACK 匹配、sync/history、事件派发 | WebSocketManager、MessageQueue |
| **WebSocketManager** | 连接建立、重连（指数退避）、帧解析、Token 传递 | - |
| **MessageQueue** | 入站批处理、出站批处理、seenIds 去重、pause/resume | - |
| **chatStore** | 认证、连接、会话、消息、UI 状态；订阅 SDKEvent 更新 | IMClient、chatPersistStorage |
| **chatPersistStorage** | StateStorage 实现，setItem 防抖，getPersistedChatState | IndexedDB |
| **ws-handler** | 帧分发、限流、db 读写、connsByUser、sendToUser | db、auth、bot |

### 3.3 WebSocket 协议（帧结构）

```typescript
// 通用帧
{ type: FrameType, seq: number, timestamp: number, payload: unknown }

// 主要 FrameType（C→S / S→C）
// C→S: SEND_MESSAGE, HEARTBEAT_PING, LOAD_HISTORY, REQUEST_AGENT, MARK_READ, ADD_REACTION, REMOVE_REACTION
// S→C: AUTH_OK, MESSAGE, MESSAGE_ACK, SESSION_SWITCHED, HISTORY_RESPONSE, READ_RECEIPT, REACTION_UPDATE, PRESENCE_UPDATE, ERROR, KICKED
```

**连接 URL**：`ws://host:3001/ws?token=<JWT>&fresh=1`。`fresh=1` 表示新建 Bot 会话，不拉历史。

### 3.4 数据库设计（SQLite）

| 表 | 主要字段 | 索引 |
|----|----------|------|
| **users** | id, address, created_at | - |
| **conversations** | id, user_id, session_type, phase, parent_conv_id, agent_id, created_at | idx_conversations_user, idx_conversations_user_type |
| **messages** | id, conversation_id, seq_id, client_msg_id, content, msg_type, sender_type, metadata, timestamp | idx_messages_conv_seq |
| **nonces** | nonce, address, expires_at | idx_nonces_expires |

**配置**：`journal_mode=WAL`、`busy_timeout=5000`、`synchronous=NORMAL`。

### 3.5 双会话模型

| 会话类型 | 前缀 | phase | 说明 |
|----------|------|-------|------|
| Bot | `conv-bot-*` | bot | 用户 ↔ Smart Assistant |
| Agent | `conv-agent-*` | queuing → agent | 转人工后新建，parent_conv_id 关联 Bot |

**转人工流程**：`request_agent` → 服务端创建 Agent 会话、phase=queuing → 模拟分配 Agent → 返回 SESSION_SWITCHED（conversationId、messages）→ 客户端 `MESSAGES_RESET` 清空并展示新会话。

## 四、状态与事件流（chatStore）

### 4.1 状态分组

| 分组 | 字段 | 触发来源 |
|------|------|----------|
| 认证 | auth, authError, authConnecting | connectWallet、connectAsGuest、KICKED |
| 连接 | connectionState, client | CONNECTED、DISCONNECTED、RECONNECTING |
| 会话 | phase, agentInfo, queue, messages, conversationId | MESSAGE_*、PHASE_CHANGED、MESSAGES_RESET、AGENT_ASSIGNED |
| 历史 | hasMoreHistory, loadingHistory | HISTORY_LOADED、loadMoreHistory |
| 辅助 | typing, onlineUsers | TYPING_*、PRESENCE_UPDATE |
| UI | isOpen, quoteTarget, scrollToInputRequest | toggleOpen、replyToMessage、sendMessage |

### 4.2 事件与 Store 更新映射

| SDKEvent | Store 行为 |
|----------|------------|
| MESSAGE_BATCH_RECEIVED | `set` 中 ids 去重，`state.messages.push({ ...m })`，Immer 拷贝避免共享引用 |
| MESSAGE_SENT / MESSAGE_RECEIVED | 单条时同样去重 + push |
| MESSAGE_STATUS_UPDATE | 查找 message 更新 status |
| MESSAGES_RESET | 清空 messages，更新 conversationId |
| REACTION_UPDATE | 支持 messageId 或 clientMsgId 匹配，更新 metadata.reactions |
| HISTORY_LOADED | prepend + 按 seqId/timestamp 排序，去重 |

### 4.3 Immer 与引用隔离

- Store 使用 `immer` 中间件，`set` 内可直接写式更新 `state.messages.push(...)`。
- **必须** `push({ ...m })` 拷贝，因 Immer 会 freeze 新 state，若与 IMClient 共享引用，IMClient 后续修改会报错。

## 五、核心难点与解决方案（详细）

### 5.1 高频消息下的渲染与 I/O 性能

**问题**：每秒数十条消息时，逐条 setState、逐条 IndexedDB 写入会导致卡顿、掉帧。

**完整数据流**：

```
WebSocket frame_in
  → WebSocketManager 解析
  → MessageQueue.enqueueIncoming(msg)
      → isDuplicate(id) ? return : seenIds.set(id), incoming.push(msg)
  → [每 50ms flush]
  → flushIncoming: batch = incoming.splice(0, 30)
  → onFlushIncoming(batch) → handleIncomingBatch
  → emit(SDKEvent.MESSAGE_BATCH_RECEIVED, batch)
  → chatStore: set(ids 去重, state.messages.push(...))
  → Zustand persist 触发 setItem
  → chatPersistStorage.setItem: 80ms 防抖，pendingValue = JSON
  → [80ms 后] flushWrite → IndexedDB put
```

**四层优化**：

| 层级 | 实现位置 | 参数 | 效果 |
|------|----------|------|------|
| MessageQueue | `IMClient` 构造：flushInterval 50, batchSize 30 | 50ms, 30 | 最多每 50ms 一次 setState，单次最多 30 条 |
| Store | MESSAGE_BATCH_RECEIVED 单次 set | - | 单次渲染替代多次 |
| persist | chatPersistStorage.setItem 防抖 | 80ms | 减少 IndexedDB put，与批处理错峰 |
| 渲染 | MessageList Virtuoso | overscan 5 | DOM 恒定 ~20，与总消息数无关 |

**为何 50ms**：人眼对 50ms 内的延迟不敏感，但能显著减少 React 渲染次数。20ms 批处理收益小，100ms 用户会感觉「消息有延迟」。

---

### 5.2 消息去重与顺序保证

**去重**：MessageQueue 维护 `seenIds: Map<string, number>`，key 为 message.id，value 为 timestamp。`cleanupDedup` 定期删除超过 5s 的条目，控制内存。

**顺序**：服务端 `seq_id` 每会话单调递增；`getMessagesAfter(convId, afterSeqId)` 按 seq 拉取；客户端 `HISTORY_LOADED` 时按 `seqId ?? timestamp` 排序。

**幂等**：服务端 `insertMessages` 同一 client_msg_id 只插入一次；ACK 用 client_msg_id 匹配乐观消息，替换为 server id。

**断线重连**：auth_ok 若服务端返回 messages 为空，客户端调用 `getPersistedMessages(convId)` 从 IndexedDB 读取兜底。补拉用 `getMessagesAfter`，afterSeqId 取本地已确认消息的最大 seq。

---

### 5.3 弹层选择器与 overflow 裁剪

**问题**：EmojiPicker、StickerPicker、MessageReactions 在 MessageList 内，父级 `overflow: hidden` 导致选择器被裁剪。

**方案**：`ReactDOM.createPortal(children, document.body)`，子组件用 `position: fixed`，通过 `anchorRef.current.getBoundingClientRect()` 计算 top/left。

**滚动同步**：MessageReactions 的气泡随列表滚动，picker 需监听列表容器的 scroll 事件，在 onScroll 时重新计算位置并 setState 更新 picker 坐标。

---

### 5.4 乐观更新与 ACK 匹配

**引用回复**：

- `sendMessage` 时若有 `quoteTarget`，构造 `metadata.quote: { messageId, senderName, content, type, timestamp }`，`quoteTarget` 可能只有 client_msg_id。
- 乐观消息立即 push 到 messages，ACK 时用 client_msg_id 匹配，替换 id。

**消息反应**：

- `addReaction`：先 `set` 修改 `metadata.reactions[emoji].push(userId)`，再 `client.addReaction`。
- 服务端 `getMessage(convId, msgId)` 支持 msgId 为 client_msg_id，查 `WHERE id = ? OR client_msg_id = ?`。
- REACTION_UPDATE 下发时带 messageId、clientMsgId，客户端用任一匹配。

---

### 5.5 虚拟列表与 flex 布局

**Virtuoso 原理**：padding-top 撑开上方、padding-bottom 撑开下方，中间只渲染 `[startIndex, endIndex]`。基于 offsetTree + ResizeObserver 处理可变高度。

**历史页高度为 0**：flex 子元素默认 `min-height: auto` 不收缩。需 `.history-main`、`.history-list-wrap` 设 `flex: 1; min-height: 0`，内部 `.history-message-list` 用 `position: absolute; inset: 0` 填满，Virtuoso 获得确定高度。

---

### 5.6 持久化方案（chatPersistStorage）

- 实现 `getItem`、`setItem`、`removeItem`，符合 Zustand persist 的 StateStorage 接口。
- `setItem`：pendingKey/pendingValue + 80ms 防抖，超时后 flushWrite 一次性 put。
- `getItem` 前：若 pendingValue 存在，先 clearTimer 并 await flushWrite，再读取，保证读一致性。
- `getPersistedChatState`：供 IMClient auth_ok 空消息时调用，返回 `{ messages, conversationId }`，仅当 conversationId 匹配时使用。

## 九、面试官追问及标准答案

### Q1：为什么批处理用 50ms，不是 20ms 或 100ms？

**答**：50ms 是经验值，在 50–100ms 区间权衡。20ms 太短，批处理效果弱，setState 次数仍然多；100ms 以上用户会感觉消息有延迟，客服场景对实时性较敏感。实际可做 A/B 测试，根据业务调参。核心是「单次渲染替代多次」的收益要大于间隔带来的延迟感。我们选 50ms 也是参考了 React 的 concurrent 调度粒度（约 5ms 一帧），50ms 约 10 帧，是一个比较稳妥的批量窗口。

---

### Q2：断线重连怎么保证不丢消息？

**答**：三端配合。① 服务端：每条消息有 seq_id 单调递增，支持 `getMessagesAfter(convId, afterSeqId)` 补拉；② 客户端：重连后 auth_ok 或 SYNC 带上 afterSeqId（取本地「最后已确认消息」的最大 seq）；③ 本地 IndexedDB：auth_ok 若服务端返回空消息（如新会话），则调用 `getPersistedChatState`，当 conversationId 匹配时用本地 messages 兜底展示，避免白屏。补拉时按 seq 合并，seenIds 去重。难点是 afterSeqId 的确定——若有未 ACK 的乐观消息，这些消息没有服务端 seq，需只对「已 ACK 的消息」取最大 seq 作为补拉起点。

---

### Q3：为什么用 Zustand 不用 Redux？

**答**：IM 场景状态更新非常频繁，Zustand 的 API 更轻量，没有 action/reducer 的样板代码，直接 `set` 即可。persist 中间件配置简单，`createJSONStorage(() => chatPersistStorage)` 一行接入自定义存储。Immer 中间件让 `set((state) => { state.messages.push(...) })` 写式更新更自然。Redux 更适合状态机复杂、需要时间旅行调试、严格单向数据流的场景。这里状态结构相对扁平，Zustand + Immer 足够，且包体积更小。

---

### Q4：Portal 用在哪些地方？为什么必须用？

**答**：EmojiPicker、StickerPicker、MessageReactions 的选择器。因为这些组件在消息列表内渲染，父级通常有 `overflow: hidden`（为了滚动性能、避免列表溢出），选择器会超出视口被裁剪。用 `createPortal(children, document.body)` 渲染到 body，用 `position: fixed` + `getBoundingClientRect()` 计算定位，脱离父级 overflow。滚动、resize 时需要重新计算坐标，否则会错位。MessageReactions 的气泡在列表内，滚动时气泡位置变化，需监听 scroll 并更新 picker 的 top/left。

---

### Q5：IndexedDB 读写一致性问题怎么处理？

**答**：写入是防抖 + 批量，有 pendingKey、pendingValue、flushTimer。在 `getItem` 或 `getPersistedChatState` 之前，若 pendingValue 不为空，先 `clearTimeout(flushTimer)` 并 `await flushWrite()`，再读取，保证读到的数据包含内存中尚未落库的部分。否则会出现「刚收到的消息在 UI 显示，但 getItem 读到的是旧快照」的不一致。这是典型的 write-through 与 read-your-writes 的权衡。

---

### Q6：双会话切换时，用户连续发消息怎么处理？

**答**：关键是明确「转人工请求发出后、SESSION_SWITCHED 到达前」这段窗口期内的消息归属。我们的策略是：request_agent 发出后，服务端创建 Agent 会话、phase 变为 queuing，此时客户端仍持有原 Bot 的 convId；SESSION_SWITCHED 到达后客户端才切换 conversationId。因此窗口期内的 SEND_MESSAGE 仍会发往 Bot 会话（服务端按 conn 的 convId 路由）。若产品希望这些消息进 Agent，需要在服务端 request_agent 处理中「锁定」该用户，将后续消息路由到新建的 Agent 会话，并在 SESSION_SWITCHED 中一并返回。当前实现是窗口期消息进 Bot，实现简单、语义清晰。

---

### Q7：限流为什么用滑动窗口而不是令牌桶？

**答**：当前是单机，滑动窗口实现简单，精度也够。按 userId 存最近 1 秒内的时间戳数组，每次发送前 filter 掉超 1 秒的，再判断 length。多实例时需要用 Redis + Lua 做分布式限流（如 ZREMRANGEBYSCORE + ZCARD + ZADD）。令牌桶允许短时突发（如 1 秒内前 100ms 发 20 条），适合「偶尔 spikes」的场景；滑动窗口更严格，适合「严格 QPS」的防刷。客服场景 20 条/秒对正常用户足够，选滑动窗口即可。若业务需要「平均 20 条/秒但允许短时突发」，可改用令牌桶。

---

### Q8：为什么 SQLite 不用 PostgreSQL？

**答**：当前是单机部署，SQLite + WAL 模式足够，部署简单、无需额外服务。better-sqlite3 是同步 API，代码简洁。索引按 conversation_id + seq_id、user_id 等建好，查询性能没问题。扩展多实例时，需要 PostgreSQL 做多写，以及 Redis 做 connsByUser、rateLimitMap 等共享状态。这是架构演进路径，不是一开始就必须的。很多 IM 系统早期用 SQLite 快速验证，后期再迁移。

---

### Q9：虚拟列表的 overscan 为什么是 5？

**答**：overscan 是视口外多渲染几条，减少快速滚动时的空白。5 是在「DOM 数量」和「滚动流畅度」之间的平衡。太小（如 1–2）快速滚动容易闪白，用户会看到空白再填充；太大（如 20）DOM 增多，抵消虚拟化收益。一般 5–10 是常见取值。可根据消息平均高度和滚动速度调参：消息越高，overscan 可以略小；滚动越快，overscan 可略大。

---

### Q10：client_msg_id 和 server id 的映射，服务端怎么存？

**答**：消息表有 `id`（服务端生成，如 `msg-${Date.now()}-${random}`）和 `client_msg_id`（客户端生成，发送时 msg.id 即为 client_msg_id）。ACK 时服务端返回 `{ clientMsgId, serverMsgId, seqId }`，客户端用 clientMsgId 匹配乐观消息，替换 id。服务端按 `conversation_id + client_msg_id` 可唯一确定一条消息，若要做防重复可建 `UNIQUE(conversation_id, client_msg_id)`。我们直接在 insert 时写入 client_msg_id，查询时 `WHERE conversation_id = ? AND (id = ? OR client_msg_id = ?)` 支持双 id 查找。

---

### Q11：MessageQueue 的 seenIds 会无限增长吗？

**答**：不会。`cleanupDedup` 在每次 enqueueIncoming 后调用，删除 `timestamp < Date.now() - deduplicationWindow` 的条目，window 默认 5 秒。所以 seenIds 最多保留 5 秒内的 id，在高频场景下大约几百到几千条，内存可控。若 5 秒内同一 id 重复出现（如网络重试、多端同步），会被过滤。

---

### Q12：Bot 回复和用户消息怎么保证顺序？会乱序吗？

**答**：服务端在同一个 `db.insertMessages` 事务内插入用户消息和 Bot 回复，seq_id 连续（用户 seqId，Bot seqId+1）。先 ACK 用户消息，再 MESSAGE 下发 Bot 回复。客户端按 seqId 排序展示，不会乱序。若未来 Bot 回复改为异步（如调 LLM API），需在服务端维护「等待回复」的状态，回复到达时再分配 seq_id 并下发，此时可能出现用户连续发多条、Bot 回复交错，需要在协议上支持「回复指向某条用户消息」的关联。

---

### Q13：如何保证多端同步时消息不重复？

**答**：① 服务端同一消息只入库一次，推送给该用户所有在线连接（sendToUser）；② 各端 MessageQueue 用 seenIds 按 message.id 去重；③ Store 的 MESSAGE_BATCH_RECEIVED、HISTORY_LOADED 等处理都用 `ids.has(m.id)` 过滤，避免重复 push。多端场景下，用户在同一设备发消息，服务端推送给该用户所有设备，各设备收到同一 msg 一次，seenIds 去重后只展示一次。若网络导致重推，seenIds 同样能过滤。

---

### Q14：为何 chatStore 和 chatSessionStore 要分开？

**答**：业务域不同。chatStore 是「客服对话」场景，WebSocket 长连接、服务端同步、Bot/Agent 双会话；chatSessionStore 是「好友/群组」场景，Mock 数据、多会话切换、无服务端。分开后路由 `/` 与 `/chat-session` 互不影响，组件按需引入各自 store，避免无关状态变化触发重渲染。若合在一起，chatSession 的 Mock 消息更新会连带触发 chat 相关组件的订阅，增加无效渲染。

---

### Q15：如果要在现有架构上支持「群聊」，需要改哪些？

**答**：① 数据模型：conversations 增加 type（bot/agent/group），messages 的 conversation_id 指向群；② 连接模型：当前 conn 绑定 convId，群聊需支持多会话订阅，可改为 conn 绑定 userId，服务端按「用户所在群」推送；③ 协议：MESSAGE 帧的 target 从单会话变为群 ID，需考虑未读、@提醒等；④ 多端：sendToUser 变为 sendToGroupMembers，按群成员列表推送；⑤ 限流：可从「单用户 QPS」扩展为「单用户单群 QPS」或「单群总 QPS」。核心是「会话」从 1:1 变为 1:N（用户-群），推送逻辑从点对点变为广播。

---

### Q16：你的底层 SDK 架构是怎么设计的？构建 IM 需要哪些能力？

**答**：SDK 分三层。① **WebSocketManager**：负责连接建立、断线重连（指数退避）、帧的解析与派发，不关心业务；② **MessageQueue**：生产者-消费者模式，入站批处理+去重、出站批处理+ACK 超时+rollback，是高频场景的核心；③ **IMClient**：组装上面两个，做会话管理、消息收发、sync/history、事件派发，对外暴露事件给 Store。构建 IM 需要：连接管理、协议定义、可靠投递（ACK、重试、rollback）、去重与顺序、离线恢复、限流。我们这些都在 SDK 和 server 里实现了，UI 只订阅事件，不碰底层逻辑。这样分层后，将来换 UI 框架或接不同后端，SDK 可以复用。
